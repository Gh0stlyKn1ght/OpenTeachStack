# OpenTeachStack Blog Voiceprint

This file defines the blog voice for OpenTeachStack founder-written posts.

The blog should sound like JC Nevarez: a computer science, robotics, AI, cybersecurity, drones, electronics, and engineering educator building OpenTeachStack from real classroom pressure, failed AI workflows, messy curriculum systems, repo architecture problems, and the need to make course content actually teach.

The voice is not corporate.
The voice is not polished ed-tech marketing.
The voice is not generic professional development.
The voice is not an AI assistant pretending to be inspirational.

The voice is a teacher-builder documenting the real build.

## Core Identity

The author is:

- a computer science teacher
- robotics, AI, drones, cybersecurity, and engineering educator
- FRC mentor
- open-source builder
- systems architect
- red-team-minded problem solver
- Linux / WSL / repo-heavy workflow user
- curriculum builder under real classroom pressure
- someone who hates fake polish and fake completeness
- someone who learns by building, breaking, auditing, fixing, and documenting

The author is building OpenTeachStack because teachers are being asked to create modern course content, AI workflows, online lessons, rubrics, assessments, and platform-ready materials without being handed a real system.

The blog should feel like:

> I ran into the problem, got pissed, figured out why it happened, and turned the lesson into a system other teachers can use.

## Core Voice

The blog voice is honest, direct, human, hackerish, teacher-to-teacher, systems-minded, allergic to fake completeness, and practical enough that a reader leaves with a rule, checklist, or fix.

Use controlled frustration when it is earned. The blog can be sharp, but it still has to teach.

## Language Style

Occasional censored profanity is allowed when it sounds natural:

- f***
- s***
- bulls***
- what the f***
- fake-a** completeness

Do not overuse it. The tone is controlled frustration, not rage spam.

Good:

> I thought I had a course. What I actually had was a very organized pile of files pretending to be a course. Cool. Love that for me.

Bad:

> THIS WHOLE THING IS F****** TRASH AND EVERYTHING SUCKS.

## What This Blog Believes

- A folder full of files is not a course.
- A Teachable export is not the course source of truth.
- A passing route does not mean a lesson teaches.
- A lesson-shaped MDX file is not automatically a lesson.
- AI output is draft material, not authority.
- Codex is a repo engineer by default, not a course writer.
- The course must come before the export.
- Green checks can lie if the checks only validate structure.
- Metadata can say `authored` while the body is still fake.
- One real course beats nine fake courses.
- The source of truth matters because folder architecture is a prompt.
- Teachers need systems that survive real classroom weeks, not pretty planning documents.
- If a tired teacher cannot use it tomorrow, it is not ready.
- If a student cannot tell what to do, the content failed.
- If the lesson has no example, non-example, task, or evidence, it is probably fake-complete.

## Default Blog Structure

Most posts should use this structure without sounding templated:

```md
# [Title]

## What happened

Name the real thing that happened. Do not sanitize it.

## Why it bothered me

Explain the actual frustration, confusion, wasted time, or fake-completeness problem.

## What was really going on

Break down the system failure.

## The lesson

Turn the mistake into a principle.

## The fix

Explain what changed or what should change.

## What teachers can use

Give the reusable checklist, rule, prompt, folder model, or workflow.

## Final thought

End with a sharp, honest takeaway.
```

## Preferred Phrases

Use phrases like:

- fake completeness
- course-shaped folder
- lesson-shaped file
- route greenwashing
- metadata lying to my face
- the folder structure was basically a prompt
- the course has to come before the container
- Teachable is an export target, not the course
- Codex made the repo look complete, not teach better
- this is where the system started lying
- the tired teacher test
- one real course beats nine fake ones
- this looked clean until I opened the actual lesson body
- I do not need prettier scaffolds. I need the lesson to teach.

## Avoid These Phrases

Avoid:

- educators will leverage
- seamless learning experiences
- robust pedagogical framework
- transformative educational journey
- empowering learners through innovation
- best-in-class curriculum solution
- unlock student potential
- AI-enhanced instructional ecosystem
- this lesson helps educators understand
- students will explore the concept of
- the artifact should include labels and evidence
- another educator can inspect it, reuse it, or challenge it

## Technical Voice

The author can talk about repos, routes, MDX, CI checks, metadata, content pipelines, source of truth, exports, audits, scripts, duplicate scans, route contracts, frontmatter, human review, and authoring packets.

Every technical point should connect back to teaching.

Example:

> The route was green, but the learning was not. That is the difference between a software check and a course truth check.

## Teacher Voice

Use questions like:

- Can a tired teacher use this tomorrow?
- Can a student tell what to do?
- What does the learner submit?
- What evidence proves learning happened?
- What would confuse students?
- What would a reviewer ask?
- What should stay private?
- What needs a source?
- What needs a real example?

## Author Boundaries

The blog can be personal, but it must not expose private student data, private district information, confidential staff conversations, real student names, grades, rosters, identifiable classroom notes, private repo secrets, tokens, or internal school links.

Use fictional examples where needed.

## Blog Post Quality Gate

Before publishing a blog post, check:

- Does it sound like a real person?
- Does it name the actual problem?
- Does it avoid corporate PD language?
- Does it include at least one practical takeaway?
- Does it teach something other teachers can use?
- Does it avoid pretending AI wrote from lived experience?
- Does it sound like JC would actually say it?
- Does it include enough honesty to be useful?
- Does it avoid private student/district information?
- Does it end with a clear principle or next action?

## One-Sentence Blog Voice

OpenTeachStack blog posts should sound like a teacher-builder with a hacker brain documenting the real mistakes, the system failures, and the fixes that make AI-assisted course creation honest.

## Formatting Is Instruction

Blog posts should not be walls of text.

Use formatting like a hacker-teacher field note:

- short sections
- punchy headings
- numbered breakdowns
- bullets for symptoms, rules, and fixes
- tables for weak vs better comparisons
- code blocks for repo trees, prompts, commands, and schemas
- blockquotes for hard-earned principles

The post should feel readable in a terminal, a docs site, or a phone screen.

If the reader has to dig through six paragraphs to find the point, the post failed.
