## What exists

- Authored draft lessons for chapters `00-13`

## Course promise

By the end, educators understand the software, vocabulary, files, workflows, AI tools, documentation habits, and safety boundaries used to build modern curriculum projects.

## Strongest parts

- The course has a clear on-ramp role before OTS-101, OTS-240, OTS-301, and OTS-320.
- Each chapter produces a practical artifact for the Teacher Builder Starter Kit.
- The course avoids turning beginners into software engineers before they understand the workflow.
- Safety, secrets, student data, backups, diffs, and AI access boundaries are introduced early enough to matter.
- The sample packet gives teachers a concrete model without using real student data.

## Review risks

- Some tool descriptions may need a technical accuracy pass against current official documentation.
- The course may still feel long for a true beginner, even though each chapter is short.
- The language should be checked for accidental developer assumptions, especially in Node, npm, deployment, GitHub, and MCP sections.
- The public pathway and book index show Draft; reviewers should confirm the visible route does not imply release readiness.
- The course has not had a browser pass for route rendering, navigation, or MDX component behavior.

## Human review questions

1. Does OTS-000 feel like a friendly on-ramp, or does it still feel too technical?
2. Are the chapter artifacts useful enough to keep, or should any be merged?
3. Does the Teacher Builder Starter Kit feel like a real final product?
4. Are any software claims too specific for a course that should stay durable?
5. Are AI-agent and MCP warnings clear without making the tools sound forbidden?
6. What must change before OTS-000 moves from public draft to live/release-ready?

## Suggested review path

1. Read chapters `00`, `01`, `04`, `07`, `11`, `12`, and `13` first.
2. Review the templates next.
3. Review the completed sample packet.
4. Check source references for current official documentation.
5. Decide whether to keep, shorten, or merge any chapters.
6. Run route/render checks only after content review approves the draft direction.
