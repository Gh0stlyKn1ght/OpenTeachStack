## The Teacher Problem

A teacher may hear, "Just use your computer," as if all computers support the same work.

They do not.

A Chromebook, Windows laptop, MacBook, and Linux machine can all be useful, but they do not give the same local tool access by default.

## The Idea

Think in terms of fit.

| Device path | Usually good for | Watch out for |
|---|---|---|
| Chromebook | browser tools, Google Workspace, GitHub reading, LMS work | local development may be blocked or limited |
| Windows | VS Code, Node, Git, browser work, WSL option | school permissions, path confusion, setup choices |
| macOS | strong local development support, terminal, VS Code | cost, permissions, Apple-specific setup details |
| Linux | strong developer workflow and servers | steeper setup if unfamiliar |

No device is morally better. The question is what job it can do safely.

## Classroom Scenario

A teacher can write Markdown in GitHub on a Chromebook and review course pages in the browser. The same teacher may need a Windows or Mac machine to run a local Next.js preview.

Both workflows are valid. They are just different.

## Weak Version

```text
I need a developer laptop before I can learn anything in this course.
```

That overstates the requirement.

## Better Version

```text
I can start with browser-based learning now, and I can identify which later tasks require local tools.
```

This keeps the on-ramp open.

## Build Step

In your checklist, add a device row:

```text

```

## Quality Check

Your row should name what the machine can do now, not just what it cannot do.

## Safety / Accessibility / Source Note

Do not bypass school device restrictions to install development tools. If the device is managed, ask what is allowed and keep student data out of personal experiments.

## Reflection

Is your main device better suited for browser-based work, local project work, or both?
