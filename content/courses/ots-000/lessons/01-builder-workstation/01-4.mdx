## The Teacher Problem

A teacher setup can fail in small annoying ways:

- the browser is outdated
- the device is locked down
- VS Code is not installed
- Node is missing
- storage is full
- files sync to the wrong account
- nobody knows whether the work is local or published

A setup checklist prevents those problems from hiding until the worst moment.

## The Idea

The Builder Workstation Checklist is a practical inventory of what your machine can do now and what needs help before advanced work.

It is not a shopping list. It is a readiness map.

## Classroom Scenario

A teacher plans to build a small course site over the weekend. Before starting, the checklist reveals that the school laptop blocks software installs.

That changes the plan: use GitHub web editing for now, then schedule local setup on an approved machine.

## Weak Version

```text
Computer: laptop
Browser: Chrome
Ready: probably
```

This does not help when something breaks.

## Better Version

| Area | Current state | Ready? | Next action |
|---|---|---|---|
| Device | school-managed Chromebook | partly | browser workflows only |
| Browser | Chrome current | yes | keep updated |
| Local installs | blocked | no | ask IT or use another machine |
| GitHub | account created | yes | add recovery email |
| Backups | Drive sync | partly | confirm project folder backup |

## Build Step

Create your checklist with these sections:

1. Device and operating system
2. Browser
3. Accounts
4. Install permissions
5. Storage and RAM
6. Local tools
7. Browser tools
8. Backup method
9. School restrictions
10. Questions to ask before advanced work

## Quality Check

Your checklist should answer:

- What can I do today?
- What cannot happen on this machine yet?
- What needs permission, help, or another setup?
- What should not touch student data?

## Safety / Accessibility / Source Note

Keep passwords and API keys out of the checklist. Write `password manager` or `stored in approved account system` instead of recording secrets.

## Reflection

What setup problem can you prevent now instead of discovering during a project deadline?
