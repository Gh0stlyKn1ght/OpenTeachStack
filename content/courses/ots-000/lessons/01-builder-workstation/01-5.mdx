## The Teacher Problem

A setup checklist can become wishful thinking.

Teachers sometimes write what they hope the machine can do instead of what they have actually confirmed.

That causes trouble later when a course build, AI agent, or publishing step assumes tools that are not available.

## The Idea

This checkpoint asks for honest readiness, not perfect readiness.

A useful answer might be:

```text
Ready for browser work now. Not ready for local builds until VS Code, Git, and Node are installed on an approved machine.
```

That is better than pretending.

## Classroom Scenario

A teacher wants to start OTS-320 later. The checklist shows they still need GitHub basics, VS Code, and a backup plan.

That tells them to finish OTS-000 and OTS-240 before using coding agents on real projects.

## Weak Version

```text
I think this computer is fine.
```

Fine for what?

## Better Version

```text
This computer is ready for browser research, GitHub reading, and Markdown drafting. It is not ready for local preview or coding-agent workflows yet.
```

This is actionable.

## Build Step

Mark each checklist section:

- Ready now
- Ready with help
- Not needed yet
- Blocked
- Unknown

For every blocked or unknown item, write one next action.

## Quality Check

Your checklist should not have vague final labels like `good` or `bad`. Use workflow labels:

- browser-ready
- writing-ready
- GitHub-review-ready
- local-preview-ready
- agent-workflow-ready

## Safety / Accessibility / Source Note

If the machine is shared, school-managed, or used with student information, add a privacy note before installing tools or connecting AI services.

## Reflection

What is the safest next step for your setup: learn more, ask permission, install a tool, or use a browser-based path for now?

## Capstone Connection

Save the revised checklist. It will combine with your vocabulary map, documentation links, safety checklist, and starter Markdown lesson in the final Starter Kit.

## Checkpoint Decision Language

Use this checkpoint to make a plain decision about **Chapter Checkpoint**:

- **Ready** means the artifact can be used by another teacher without a side explanation from you.
- **Revise** means the idea is right, but the artifact still needs clearer steps, labels, evidence, or safer wording.
- **Blocked** means you cannot honestly finish the artifact yet because a tool, source, permission, example, or decision is missing.

## Evidence to Save

Save evidence that another teacher could inspect: a screenshot, copied command output, file path, short reflection note, or before-and-after excerpt. If the evidence is only in your memory, the checkpoint is not ready yet.

## Next Revision Trigger

Name the owner and trigger before you move on. Write who will revise it, when they will revisit it, and what would cause the next revision: a broken link, confusing student direction, missing source, unsafe permission, or a teammate review note.
