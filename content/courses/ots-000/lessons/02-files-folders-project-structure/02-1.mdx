## The Teacher Problem

Project instructions often assume file vocabulary.

They say:

```text
Open content/courses/ots-101/lessons/02-standards-goals-to-lessons/02-4.mdx.
```

To a beginner, that can look like a wall of slashes.

## The Idea

Use plain meanings first.

| Word | Plain meaning | Example |
|---|---|---|
| File | One saved item | `02-4.mdx` |
| Folder | A container that holds files or folders | `lessons` |
| Path | The address to a file or folder | `content/courses/ots-101` |
| Extension | The ending that hints at file type | `.mdx`, `.json`, `.tsx` |

The path tells you where something lives. The extension gives you a clue about what kind of file it is.

## Classroom Scenario

A teacher sees two files:

```text
lesson-map.md
lesson-map.pdf
```

They look related, but they are not the same kind of file. The Markdown file is easier to edit as source. The PDF is usually an export for reading or sharing.

## Weak Version

```text
The names match, so either file is fine.
```

That can lead to editing an export and wondering why the source never changes.

## Better Version

```text
The file extension helps me decide whether this is source, export, config, media, or code.
```

## Build Step

Add this section to your project-folder map:

```text
Path I found:
Filename:
Extension:
What I think it is:
Should I edit it? yes / no / not sure
Why:
```

Use three files.

## Quality Check

If your answer is `not sure`, that is a valid answer. Marking uncertainty is safer than guessing.

## Safety / Accessibility / Source Note

Be careful with files that include student names, grades, screenshots, exports, or private notes. File names can expose information even before the file is opened.
