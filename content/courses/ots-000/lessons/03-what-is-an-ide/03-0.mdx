> OTS-000 frame: an IDE is not proof that you are coding. It is a workbench for seeing a project clearly.

## The Teacher Problem

Many teachers first see VS Code or another IDE during a stressful moment.

The screen has sidebars, tabs, icons, colors, terminals, warnings, extensions, and source control panels.

It can feel like a cockpit.

## The Idea

An IDE is an integrated development environment. Plain version: it is a project editor with extra tools built in.

For teacher-builders, the useful parts are:

- file explorer
- search
- tabs
- editor area
- preview tools
- terminal panel
- source control panel
- extensions

You do not need to use every button.

## Classroom Scenario

A teacher opens a course project in VS Code. They only need to find one Markdown file, search for a title, edit a sentence, and preview the change.

The IDE has many features, but the teacher needs a small path through it.

## Weak Version

```text
VS Code is for developers, so I should avoid it.
```

That blocks a useful tool.

## Better Version

```text
VS Code is a project workbench. I can learn the few areas I need before touching anything risky.
```

## Build Step

Start an IDE navigation diagram.

Draw or list these areas:

- Explorer
- Editor
- Tabs
- Search
- Terminal
- Source Control
- Extensions

Leave space to add what each area is for.

## Quality Check

Your diagram should help someone find where to open a file, search text, run a command, and check changes.

## Safety / Accessibility / Source Note

Do not install random extensions just because they look helpful. Extensions can change behavior, read files, or add account connections.
