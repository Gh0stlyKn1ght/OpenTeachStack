## The Teacher Problem

An IDE looks complicated partly because all the tools are visible at once.

A teacher does not need to learn everything first. They need to know which area does which job.

## The Idea

Start with the main areas.

| Area | What it does | Teacher use |
|---|---|---|
| Explorer | shows project files and folders | find and open course files |
| Tabs | show open files | avoid editing the wrong file |
| Editor | where file content appears | read and revise content |
| Search | finds text across the project | locate titles, links, terms, or stale wording |
| Terminal | runs typed commands | preview, install, test, or build |
| Source Control | shows changed files | review what changed before committing |
| Extensions | add features | Markdown preview, linting, AI tools, themes |

## Classroom Scenario

A teacher needs to change a phrase used in multiple lessons. Opening files one by one is slow. Search can find every occurrence, but Source Control should still be checked before accepting the edits.

## Weak Version

```text
I found the search result, so I changed every match.
```

Search finds text. It does not decide whether every match should change.

## Better Version

```text
I used Search to find candidates, edited only the right files, then checked Source Control to review the changed files.
```

## Build Step

Add each IDE area to your navigation diagram.

For each area, write:

```text
I use this when...
I should be careful because...
```

## Quality Check

Your diagram should make it clear that Search helps find, Terminal helps run, and Source Control helps review. Do not blend those jobs together.

## Safety / Accessibility / Source Note

Before running terminal commands, check the current folder. Before accepting source-control changes, check whether unrelated files changed.

## Reflection

Which IDE area would most help you avoid mistakes: Search, Tabs, Terminal, or Source Control?

## Framework Purpose

This framework exists to keep **Explorer, Tabs, Search, Terminal, and Source Control** from turning into a pile of tips. A teacher should be able to use it when the work feels fuzzy: name the parts, make the next move smaller, and check whether the result is actually useful in a classroom.

## How to Use This Framework

Apply the framework in three passes:

1. Name the classroom job this tool or idea is supposed to do.
2. Match each part of the framework to one visible choice in your project.
3. Review the artifact for the obvious failure mode: polished wording with no evidence, unclear steps, missing source, or a tool choice that creates extra work for students or teachers.

Save the result in your Starter Kit as evidence of the decision, not as decoration.
