## The Teacher Problem

Beginners often see color-coded text and extension prompts and assume something technical is happening beyond their reach.

Sometimes the color is just a visual cue. Sometimes an extension is useful. Sometimes an extension is clutter.

## The Idea

Syntax highlighting colors parts of a file so structure is easier to see.

In Markdown, headings may look different from links. In JSON, keys and values may appear in different colors. In code, strings, functions, and comments may be visually separated.

Extensions add features to the editor.

Useful extensions might help with Markdown preview, spell check, formatting, Git history, or AI assistance.

## Classroom Scenario

A teacher opens an MDX lesson file and notices the frontmatter at the top is colored differently from the lesson body.

That visual difference helps the teacher avoid deleting metadata accidentally.

## Weak Version

```text
The colors mean this is code, so I should not touch it.
```

Not always. Some colored files are teacher-editable content.

## Better Version

```text
The colors help me see file structure. I still need to know whether this file is safe to edit.
```

## Build Step

Add two sections to your IDE diagram:

```text
Syntax highlighting helps me notice:

Extensions I might need:
Extensions I should avoid for now:
```

Keep the extension list short.

## Quality Check

A beginner extension list should not be longer than the teacher's actual workflow. Start with what helps you read, write, search, and preview.

## Safety / Accessibility / Source Note

Extensions can request access to files, accounts, terminals, or remote services. Install from trusted sources and avoid adding extensions you do not understand.

## Reflection

Which visual cue would help you most: headings, links, frontmatter, errors, or changed files?

## Capstone Connection

Your Starter Kit should include a short extension policy: what you use, why you use it, and what you will not install casually.
