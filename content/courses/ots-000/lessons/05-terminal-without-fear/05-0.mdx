## The Teacher Problem

The terminal looks unforgiving.

Tutorials paste commands with almost no explanation, and beginners are left wondering whether pressing Enter will preview a site, install a tool, delete a folder, or publish something.

That uncertainty is reasonable.

## The Idea

The terminal is a place to run typed commands in a specific folder.

For teacher-builders, the first rule is simple: know where you are before you run anything.

## Classroom Scenario

A teacher copies `npm run dev` from a README. It works only if the terminal is inside the project folder with the right package file.

The command is not magic. It depends on location.

## Weak Version

```text
The tutorial says to run this, so I will paste it anywhere.
```

That is how command fear becomes command danger.

## Better Version

```text
Before I run the command, I will check the current folder and read what the command is supposed to do.
```

## Build Step

Start a terminal command cheat sheet with these columns:

```text
Command | Plain meaning | Where to run it | What it changes | Stop if
```

## Quality Check

Every command should include a plain-language meaning. If you cannot explain it, do not run it yet.

## Safety / Accessibility / Source Note

Do not paste terminal commands from random sources into school or student-data projects. Use official docs, project READMEs, or trusted guidance.

## Reflection

What makes terminal commands feel risky to you: the words, the black screen, the unknown effect, or the error messages?

## Capstone Connection

The terminal cheat sheet becomes the command-safety section of your Teacher Builder Starter Kit.
