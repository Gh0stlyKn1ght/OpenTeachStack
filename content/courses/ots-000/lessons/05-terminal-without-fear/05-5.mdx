## The Teacher Problem

A command cheat sheet can create false confidence if it only tells teachers what to type.

The better question is what the teacher understands before typing.

## The Idea

This checkpoint asks whether your cheat sheet supports safe command use.

It should include location, purpose, effect, and stop conditions.

## Classroom Scenario

A teacher finds a command in a blog post. The cheat sheet gives them a decision rule: if the command is not in official docs, the README, or trusted guidance, do not paste it into a real project.

## Weak Version

```text
I copied the command correctly.
```

Correct copying is not enough.

## Better Version

```text
I know what the command does, where to run it, and what it might change.
```

## Build Step

Review your cheat sheet and mark each command:

- Reads only
- Moves around
- Creates or changes files
- Installs packages
- Starts a process
- Needs help before running

## Quality Check

At least one command should be marked "needs help before running." If nothing looks risky, you are probably missing the safety layer.

## Safety / Accessibility / Source Note

Never run commands involving secrets, tokens, student data, or production publishing unless you understand the environment and have permission.

## Reflection

What command or command type will you refuse to run casually?

## Capstone Connection

Save the revised cheat sheet. It will support GitHub and coding-agent work later in the pathway.
