## The Teacher Problem

Teachers often enter a GitHub repo by clicking files at random.

That is understandable, but it skips the project’s front door: the README and recent history.

## The Idea

Start with landmarks.

| Landmark | What to look for |
|---|---|
| README | purpose, setup, use, warnings |
| Folders | where content, docs, code, and assets live |
| Commits | what recently changed |
| Issues | known problems or requests |
| Pull requests | proposed changes under review |
| License | reuse rules |

## Classroom Scenario

A teacher wants to know whether a repo contains live course content or a stale export. The README and recent commits may answer that before any file edits happen.

## Weak Version

```text
I found a file that looks right, so I will edit it.
```

The file might be old, generated, or not the source of truth.

## Better Version

```text
I read the README, checked the folder map, and looked at recent commits before choosing what to edit.
```

## Build Step

Add a repo-reading sequence to your checklist:

1. Read the repo name and description.
2. Open the README.
3. Identify the main folders.
4. Check recent commits.
5. Look for issues or pull requests if the project is active.
6. Check the license or reuse note.

## Quality Check

You should be able to explain what the repo is for before editing any file.

## Safety / Accessibility / Source Note

If a repo is public, assume anything committed there can be seen. Do not use public GitHub as a storage place for private class materials.
