## The Teacher Problem

Software instructions are everywhere. Official docs, blog posts, videos, forum answers, AI summaries, old screenshots, and copied command snippets all compete for attention.

Teachers need a way to decide what to trust.

## The Idea

Documentation literacy means you can find the right kind of help for the task.

You do not need to read every page. You need to know whether you are looking at an install guide, quickstart, tutorial, API reference, changelog, troubleshooting note, or random opinion.

## Classroom Scenario

A teacher asks why a command from an old tutorial does not work. The official docs show the tool changed its setup instructions last year.

The teacher was not wrong. The source was stale.

## Weak Version

```text
I found instructions online, so they should work.
```

Online is not the same as current or official.

## Better Version

```text
I found the official docs, checked the version, and used the install page that matches my system.
```

## Build Step

Start a documentation evaluation checklist:

```text
Tool:
Official docs link:
Page type:
Version or date:
What I need from it:
What I still need to verify:
```

## Quality Check

The checklist should help you decide whether a source is good enough to guide a real project step.

## Safety / Accessibility / Source Note

AI can summarize docs, but it can also blur versions, omit warnings, or invent commands. Use AI for orientation, then verify with official documentation.
