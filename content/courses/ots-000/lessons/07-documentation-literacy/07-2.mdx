## The Teacher Problem

Teachers can land on the wrong docs page and think the whole tool is too hard.

Sometimes the problem is not ability. It is page type.

## The Idea

Different docs pages answer different questions.

| Page type | Question it answers |
|---|---|
| Install guide | How do I get the tool onto my computer? |
| Quickstart | What is the fastest safe first run? |
| Tutorial | How do I learn a workflow step by step? |
| Examples | What does working usage look like? |
| API reference | What exact options or functions exist? |
| Troubleshooting | What do I do when something breaks? |
| Changelog | What changed between versions? |

## Classroom Scenario

A teacher wants to know why a build fails. The API reference is probably not the first stop. The troubleshooting page, error message, or project README may be more useful.

## Weak Version

```text
I opened the docs and did not understand the page, so the tool is too advanced.
```

Maybe the page was not meant for your question.

## Better Version

```text
I will match my question to the docs page type before deciding the tool is too hard.
```

## Build Step

Pick one tool and find three docs pages:

- install or setup
- quickstart or tutorial
- troubleshooting or FAQ

Add them to your checklist with page type labels.

## Quality Check

Each link should answer a different question. If all three links do the same job, keep looking.

## Safety / Accessibility / Source Note

Install guides and command examples deserve extra care. Check operating system, version, and whether admin rights are required.

## Reflection

Which page type do you usually need first: install, quickstart, tutorial, or troubleshooting?

## Capstone Connection

Your Starter Kit will include a docs bookmark pack sorted by page type, not just a pile of links.
