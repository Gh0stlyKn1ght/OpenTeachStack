## The Teacher Problem

A bookmark folder full of links can still be useless.

If the links are not labeled by purpose, currentness, and trust level, the teacher has to re-evaluate them every time.

## The Idea

A documentation evaluation checklist helps you keep reliable links attached to a clear job.

It should show what the link is for, whether it is official, when you checked it, and what risk it carries.

## Classroom Scenario

A teacher collects links for VS Code, GitHub, Node, and Markdown. Instead of dumping them into one list, the teacher labels install pages, beginner docs, troubleshooting pages, and safety notes separately.

## Weak Version

```text
Useful links:
- VS Code
- GitHub
- Node
```

This does not tell the teacher what to do with each link.

## Better Version

| Tool | Link | Page type | Official? | Date checked | Use |
|---|---|---|---|---|---|
| VS Code | official setup page | install guide | yes | 2026-06-27 | install safely |
| GitHub | docs homepage | reference | yes | 2026-06-27 | learn repo basics |

## Build Step

Create a checklist with these columns:

```text
Tool | Link | Page type | Official? | Version/date | Why I need it | Risk or warning
```

Add at least five links for tools in OTS-000.

## Quality Check

At least three links should be official documentation. If the list is mostly videos or AI summaries, revise it.

## Safety / Accessibility / Source Note

Do not include unofficial download links. For installs, use official vendor pages or approved organization portals.

## Reflection

Which link in your list is most important to keep current?
