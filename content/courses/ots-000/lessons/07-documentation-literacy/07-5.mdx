## The Teacher Problem

A documentation checklist can still fail if it does not lead to a decision.

The teacher needs to know which links are safe to follow now and which need verification later.

## The Idea

This checkpoint asks whether your checklist separates trusted install guidance, learning support, troubleshooting, and stale-risk links.

## Classroom Scenario

A teacher is about to install a command-line tool. The checklist points to an official install page and marks a blog tutorial as "learning support only."

That prevents the teacher from installing from the wrong source.

## Weak Version

```text
I saved the link, so I am done.
```

Saving is not evaluation.

## Better Version

```text
I know what the link is for, whether it is official, when I checked it, and what risk it carries.
```

## Build Step

Review each link and label it:

- official and current
- helpful but unofficial
- outdated or version-specific
- needs verification
- remove

## Quality Check

Your list should include at least one remove or needs verification item. If every link seems equally trustworthy, you are probably not evaluating hard enough.

## Safety / Accessibility / Source Note

When a tool affects accounts, permissions, installs, commands, publishing, or student data, do not rely on a single unofficial source.

## Reflection

Which link will you verify again before using it in a real setup?

## Capstone Connection

Save the checklist. It will support the rest of the course whenever a tool or command appears.
