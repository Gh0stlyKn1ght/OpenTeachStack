## The Teacher Problem

MDX files can look like normal Markdown until a teacher sees metadata or components and worries the whole file is unsafe to edit.

The better move is to separate the lesson body from the structured parts.

## The Idea

Frontmatter is metadata at the top of a file.

It may control title, route, order, status, duration, or other information the site needs.

MDX components are special blocks inside Markdown that render custom visuals or behavior.

## Classroom Scenario

A teacher opens a lesson file and sees:

```md
---
title: "Chapter Overview"
order: 1
---

## The Teacher Problem
```

The teacher can revise the lesson body but should pause before changing `title`, `order`, or route fields.

## Weak Version

```text
I should delete the metadata because students do not need to read it.
```

That metadata may control the site.

## Better Version

```text
I will edit the lesson body and ask before changing frontmatter or components that affect rendering.
```

## Build Step

Add a frontmatter note above your starter lesson:

```md
---
title: "Starter Lesson"
status: "draft"
---
```

Then write one sentence explaining that frontmatter is metadata, not student-facing lesson prose.

## Quality Check

You should know which parts of the file are safe to revise and which parts need review.

## Safety / Accessibility / Source Note

Breaking frontmatter can break routes, titles, ordering, or publishing metadata. Do not bulk-edit frontmatter unless you have a clear map and a verification step.

## Reflection

What is one metadata field you would ask about before changing?

## Capstone Connection

Your Starter Kit should include a note that separates lesson body edits from metadata edits.
