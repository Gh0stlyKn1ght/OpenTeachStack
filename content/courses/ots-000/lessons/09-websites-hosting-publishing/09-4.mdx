## The Teacher Problem

Publishing feels like one button until something goes wrong.

Then the teacher needs the steps.

## The Idea

A publishing flow diagram shows the path from source files to student view.

It should show each step and the check that belongs there.

## Classroom Scenario

A teacher publishes a course hub. The source files are correct, the build passes, the deployment succeeds, but students cannot access one linked document.

The flow diagram shows that student-view access is a separate check after deployment.

## Weak Version

```text
Edit -> publish -> done
```

This skips too many checks.

## Better Version

```text
Source file -> local preview -> build -> deploy -> published URL -> student-view check -> revision note
```

## Build Step

Create your diagram with these boxes:

1. Source files
2. Local preview
3. Build
4. Deployment
5. Published URL
6. Student view
7. Revision note

Under each box, write:

```text
What I check here:
What can go wrong:
Evidence I save:
```

## Quality Check

The diagram should make clear that published does not mean reviewed.

## Safety / Accessibility / Source Note

Add privacy, accessibility, source, and link checks before the final student-view box.

## Reflection

Which step would you be most likely to forget under deadline pressure?
