## The Teacher Problem

Modern project names pile up fast.

A teacher may hear React, Next.js, Vite, TypeScript, Tailwind, Docusaurus, Nextra, and Vercel in the same conversation and lose the thread.

## The Idea

Use job labels.

| Tool | Plain job |
|---|---|
| React | builds interactive interface pieces |
| Next.js | framework for React sites and apps |
| Vite | fast development/build tooling often used with React |
| TypeScript | JavaScript with type checks |
| Tailwind CSS | utility-based styling system |
| MDX | Markdown plus components |
| Docusaurus | documentation site framework |
| Nextra | documentation framework built on Next.js |
| Vercel | hosting and deployment platform |

You do not need to master all of these in OTS-000. You need to recognize what layer they belong to.

## Classroom Scenario

A teacher sees an error in a `.tsx` file and an MDX lesson file nearby. The teacher can revise the lesson file but should ask for help before changing the component unless the task is clear.

## Weak Version

```text
All these tools are the same: website code.
```

Too blurry.

## Better Version

```text
Some tools control content, some control interface, some control styling, and some control building or hosting.
```

## Build Step

Add any tools your project uses to the stack card.

Use this format:

```text
Tool:
Layer:
Plain job:
Do I edit this directly? yes / no / with help
```

## Quality Check

If the tool's job is still unclear, write `unknown` and add it to your documentation checklist.

## Safety / Accessibility / Source Note

Styling and interface changes can affect accessibility. Do not treat visual changes as harmless just because they are not lesson text.

## Reflection

Which tool name became less mysterious after you gave it a job?

## Capstone Connection

The stack card helps you explain a project before asking Codex, Claude Code, Gemini, or a human helper to change it.

## Framework Purpose

This framework exists to keep **React, Next.js, Vite, TypeScript, Tailwind, Docusaurus, and Nextra** from turning into a pile of tips. A teacher should be able to use it when the work feels fuzzy: name the parts, make the next move smaller, and check whether the result is actually useful in a classroom.

## How to Use This Framework

Apply the framework in three passes:

1. Name the classroom job this tool or idea is supposed to do.
2. Match each part of the framework to one visible choice in your project.
3. Review the artifact for the obvious failure mode: polished wording with no evidence, unclear steps, missing source, or a tool choice that creates extra work for students or teachers.

Save the result in your Starter Kit as evidence of the decision, not as decoration.
