## The Teacher Problem

A stack card can sound confident while still hiding unknowns.

That is dangerous because unknown tools often carry the most risk.

## The Idea

This checkpoint asks whether your stack card separates known, unknown, safe-to-edit, and review-required areas.

## Classroom Scenario

A teacher marks MDX lesson files as teacher-editable but marks deployment settings as "review before editing."

That is a healthy boundary.

## Weak Version

```text
I listed the tools, so I understand the stack.
```

Listing is not the same as knowing the job and boundary.

## Better Version

```text
I can explain each tool's job and name what I will not edit without review.
```

## Build Step

Review each stack card row and mark:

- I can explain this.
- I need docs.
- I need human review.
- I should not edit this yet.

## Quality Check

Your card should include at least one review boundary. If everything is marked safe, the card is too loose.

## Safety / Accessibility / Source Note

Deployment, environment variables, authentication, private data, and automation workflows deserve extra caution.

## Reflection

Which tool in the stack should you learn more about before using AI help on it?

## Capstone Connection

Save the revised stack card. It becomes the bridge from basic tool literacy into AI-agent workflow decisions.
