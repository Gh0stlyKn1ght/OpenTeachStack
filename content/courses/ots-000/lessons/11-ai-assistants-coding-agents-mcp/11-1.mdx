## The Teacher Problem

AI labels are messy.

A tool might call itself an assistant, copilot, agent, chatbot, or workspace. The useful question is not the label. It is what the tool can do.

## The Idea

Use categories by access.

| Category | Typical job | Main risk |
|---|---|---|
| Chatbot | explains, drafts, brainstorms | output may be wrong or generic |
| IDE assistant | suggests code or text inside editor | may change files quickly |
| Autocomplete copilot | predicts next lines | may insert weak or unsafe code |
| Terminal/repo agent | reads files, edits, runs commands | can change many files |
| MCP-connected agent | connects to tools and services | access boundaries matter |

## Classroom Scenario

A teacher wants help explaining what an IDE is. A chatbot is fine.

A teacher wants an agent to update route files. That needs repo scope, Git safety, and review.

## Weak Version

```text
I will use the most powerful AI tool for every task.
```

Power adds risk.

## Better Version

```text
I will use the least powerful tool that can safely do the job.
```

## Build Step

Add tool categories to your map:

```text
Task | Lowest-risk AI category | Why | Review needed
```

Add three tasks from your work.

## Quality Check

If a task only needs explanation, do not start with a repo-editing agent.

## Safety / Accessibility / Source Note

AI output is draft material. Even a low-access chatbot can produce inaccurate links, citations, code, or policy claims.

## Reflection

Where have you used a more powerful AI tool than the task required?

## Capstone Connection

Your Starter Kit should include a tool-choice rule before AI enters a project.
