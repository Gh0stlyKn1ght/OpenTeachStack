## The Teacher Problem

MCP can sound abstract until something gets access.

Teachers need the simple version: MCP lets AI applications connect to tools and data through a shared protocol.

That can be useful. It also needs boundaries.

## The Idea

Plain terms:

| Term | Plain meaning |
|---|---|
| MCP client | the AI app using the connection |
| MCP server | the connector that exposes tools or data |
| Tool | an action the AI can request |
| Context | information the AI can use |
| Trust boundary | the line around what the AI is allowed to reach |

## Classroom Scenario

An AI tool connected to GitHub can inspect issues or pull requests. An AI tool connected to local files can read project folders. Those are different access levels.

## Weak Version

```text
MCP makes the AI better, so more connections are better.
```

More access is not automatically better.

## Better Version

```text
MCP gives agents access. I need to know what the connection exposes before using it.
```

## Build Step

Create an access map:

```text
Connected system:
What the agent can read:
What the agent can change:
What approval is needed:
What must stay private:
```

## Quality Check

If you cannot name what the connection can read or change, do not use it yet.

## Safety / Accessibility / Source Note

MCP access can include files, services, repos, databases, browsers, calendars, or custom tools. Treat access as a permission decision, not a feature toggle.

## Reflection

What system would you be cautious about connecting to an AI agent?

## Capstone Connection

The MCP access map becomes part of your Starter Kit and prepares you for safer advanced agent workflows.
