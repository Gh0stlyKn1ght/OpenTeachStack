## The Teacher Problem

Teachers may not think of passwords, API keys, private links, rosters, screenshots, or student examples as "secrets" until a tool asks for access.

By then, the boundary may already be blurry.

## The Idea

Secrets are values that grant access. Private data is information that should not be exposed. Both need protection.

Examples:

- passwords
- API keys and tokens
- `.env` files
- private repo links
- student names or grades
- screenshots with account details
- unpublished sensitive documents
- private school system URLs

## Classroom Scenario

A teacher wants AI help debugging a tool setup and pastes the `.env` file into chat. That file may contain keys that grant access.

The safer move is to remove secrets and describe the problem.

## Weak Version

```text
The AI needs all the details, so I will paste the config file.
```

No.

## Better Version

```text
I will remove secrets, use placeholders, and share only the minimum needed context.
```

Add a privacy boundary to your checklist:

```text

```

## Quality Check

If a piece of information would be bad on a public website, do not put it into a public repo or AI prompt.

## Safety / Accessibility / Source Note

Do not rely on deleting a message or commit as your only fix after exposing a secret. Treat exposed secrets as compromised and rotate them through the proper system.
