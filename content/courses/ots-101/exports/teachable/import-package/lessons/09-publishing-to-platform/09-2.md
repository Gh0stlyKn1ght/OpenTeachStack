> OTS-101 frame: AI supports teacher-approved course content that becomes student-facing only after review.

Before publishing, prepare the content for the platform on purpose.

Do not paste first and repair later if you can avoid it.

Start from the approved source. Then adapt the lesson to the delivery surface.

## Step 1: Confirm the source

Before copying anything, answer:

- Which file is the approved source?
- Has it passed teacher review?
- Does it include the latest safety and access checks?
- Is this the version students should see?

If you are not sure, stop and find the source first.

## Step 2: Choose the right platform surface

Different content belongs in different places.

Examples:

- lesson explanation: LMS page or website page
- student work task: LMS assignment
- quick check: quiz or form
- reference material: file or resource page
- demonstration sequence: slide deck
- offline work: printable handout

Do not force everything into one page if the platform has a better structure.

Keep this decision at the delivery level. You are not choosing hosting architecture, DNS, deployment tooling, or a custom course-site stack in OTS-101. You are choosing the student-facing surface that fits the approved content.

## Step 3: Preserve student-facing structure

Keep the student path visible:

- learning target
- why this matters
- materials
- directions
- example
- before you submit
- need help

Use real headings and lists.

If the platform editor strips formatting, rebuild it manually.

## Step 4: Attach materials carefully

Every file, link, form, or video should be checked.

Ask:

- Is it the correct file?
- Can students access it?
- Is it named clearly?
- Does it match the directions?
- Does it need attribution or context?

Do not assume the platform fixed permissions for you.

## Step 5: Add a source note

A source note can be private or visible to teachers only.

Example:

```text
Source: course-content/source/chapter-05/student-facing-lesson-approved.md
Platform export: Canvas assignment updated 2026-06-26
Reviewer: JC
```

That note helps prevent version drift.

## Ask AI for platform adaptation carefully

AI can help adapt formatting, but it should not invent missing platform settings.

Prompt:

```text
Using this approved lesson source, prepare a platform-ready version for an LMS assignment page. Preserve the learning target, directions, example, success check, and submission instructions. Do not add new links or tools. End with a teacher-check list for platform settings I must verify manually.
```

<TakeawayStrip tone="teacher">
  Platform preparation is not about making the page pretty. It is about making sure the approved lesson survives the move into the place students use it.
</TakeawayStrip>

## When to use this workflow

Use this workflow when the lesson has already passed teacher review and now needs to move into the platform students will actually use.

Do not use it to rescue an unreviewed draft. Publishing prep should preserve an approved source, not hide weak content behind platform formatting.

## Classroom scenario

A teacher has an approved lesson page and wants to put it into Canvas, Google Classroom, Moodle, or another LMS. The lesson is solid in the source file, but the platform editor strips headings, breaks a table, and hides the rubric behind a setting.

The danger is not that the lesson disappears. The danger is that the platform copy looks published while the student path is broken.

## Weak version

```text
I pasted the lesson into the LMS and attached the file. It should be fine.
```

This version trusts the paste instead of checking what students receive.

## Better version

```text
Platform preparation note:
- Source: approved lesson file from the packet source folder
- Surface: LMS assignment page
- Preserved: learning target, directions, example, success criteria, submission instructions
- Added: student-visible file link and rubric
- Manual checks still needed: permissions, due date, rubric visibility, student view
```

This version keeps the source and the platform copy connected.

## Workflow steps

1. Confirm the approved source file.
2. Choose the platform surface that matches the student action.
3. Paste or rebuild the content while preserving headings, examples, checks, and submission directions.
4. Attach only the files, links, rubrics, and media the lesson actually requires.
5. Add a private source note so the platform copy can be traced back later.
6. Mark platform settings that must be checked manually.

## Quality check

Ask:

- Did I start from the approved source?
- Did the platform keep the structure students need?
- Did every attachment, link, rubric, and file land in the right place?
- Are teacher-only notes hidden from students?
- Can I trace this export back to its source?

## AI support

AI can help reformat an approved source for a platform, but it should not invent due dates, settings, links, rubrics, or access permissions.

Ask for a platform-ready draft and a teacher-check list. Then do the platform checks yourself.

If the platform requires technical setup beyond normal teacher publishing, mark that as a later-pathway issue instead of expanding this checklist.

## Next action

Prepare one approved lesson for a real platform surface. Save the source note and the manual settings list in your platform publishing checklist.
