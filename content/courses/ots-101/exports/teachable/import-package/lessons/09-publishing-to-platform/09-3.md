> OTS-101 frame: AI supports teacher-approved course content that becomes student-facing only after review.

The teacher view can lie.

Not because the platform is trying to trick you, but because teacher accounts often have access students do not have.

A link that works for the teacher may fail for students. A file may be visible to the owner but not the class. A rubric may be attached but hidden. A due date may be missing. A module may be unpublished.

That is why student view matters.

## Start from the student path

Do not only preview the page directly.

Start where students start.

For example:

- course home page
- module page
- classwork stream
- weekly agenda
- assignment list
- linked calendar item

Ask whether the lesson is findable from the path students actually use.

## Open every required item

Click like a student.

Check:

- links
- documents
- videos
- forms
- slides
- images
- downloads
- embedded content
- rubric or success criteria

If students need it, test it.

## Compare directions to platform settings

Student directions and platform settings must match.

Common mismatches:

| Direction says | Platform actually does |
|---|---|
| Upload a file | Text entry only |
| Complete the form | Form link missing |
| Use the attached rubric | Rubric hidden |
| Submit by Friday | No due date shown |
| Work with a partner | Individual submission settings only |
| Watch the video | Video blocked or private |

Fix mismatches before release.

## Check formatting after publish

Look for the basics:

- headings are visible
- bullets are bullets
- numbered steps stay numbered
- tables are readable
- examples are separated from directions
- success check is easy to find
- long paragraphs are broken up

If the platform flattens the content, rebuild the structure.

## Add a release note

A short release note helps you remember what was checked.

```text
Student view checked:
- module path visible
- links open for students
- assignment accepts text entry
- rubric visible
- source note recorded
```

<TakeawayStrip tone="teacher">
  If the student view does not work, the lesson is not published yet. It is only placed somewhere.
</TakeawayStrip>

## When to use this workflow

Use this workflow after the platform copy is built and before students are told to use it.

The student view check is the release gate. If the student path fails, the lesson is not ready even if the teacher-facing page looks finished.

## Classroom scenario

A teacher publishes an assignment from the teacher account. The lesson page opens, the attached document opens, and the rubric appears.

In student view, the document says "request access," the rubric is hidden, and the submit button only allows text entry even though the directions ask for a file upload.

Nothing about the lesson goal is wrong. The release is wrong.

## Weak version

```text
I previewed the page from my account and everything opened.
```

Teacher preview is useful, but it is not enough.

## Better version

```text
Student-view release note:
- Started from the module page
- Assignment visible to students
- Lesson directions readable
- Document opened with student permissions
- Rubric visible
- Submission type matches directions
- Due date visible
```

This version checks the path students will actually take.

## Workflow steps

1. Enter the course from the same starting point students use.
2. Open the lesson, assignment, files, media, forms, and links.
3. Compare each student-facing direction to the platform setting.
4. Check that examples, success criteria, rubrics, and submission instructions are visible.
5. Fix mismatches before release.
6. Save a short student-view release note.

## Quality check

Ask:

- Can a student find the lesson without a private teacher link?
- Can a student open every required item?
- Can a student complete the task using only the visible directions?
- Can a student submit in the format the directions require?
- Would the page still make sense if the student missed the verbal explanation?

## Safety and access note

Student view can expose safety and equity problems the teacher view hides: blocked videos, inaccessible files, account prompts, public sharing settings, or tools that require personal data.

Treat those as release blockers, not small annoyances.

## Next action

Run one real student-view check and write the release note in your platform publishing checklist.
