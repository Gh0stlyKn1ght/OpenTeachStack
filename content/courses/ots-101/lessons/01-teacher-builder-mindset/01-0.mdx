# Chapter Plan

## Why This Matters

A teacher building a first reusable mini-unit needs a system another educator can inspect. A system is more than a folder, a prompt, a worksheet, or a favorite app. It is the connection between the learning goal, the source material, the classroom task, the student evidence, and the revision routine after the lesson is taught.

This chapter starts with the teacher as a builder because most curriculum problems do not begin with a lack of tools. They begin when useful work lives in too many places, decisions stay in the teacher's head, and no one can tell what version is ready for students. The chapter artifact, a **Teacher workflow audit**, helps you name one teaching workflow and make its moving parts visible.

## Chapter Situation

Use one real or realistic mini-unit as your working example. The model in this chapter is a middle school robotics mini-unit where students use an ultrasonic sensor, record distance readings, explain measurement uncertainty, adjust a threshold, and write a short design reflection.

The content area can change. The workflow questions stay the same:

- What source or standard tells the teacher what matters?
- What will students actually do during class?
- What product, response, or observation will show learning?
- Where will the teacher store the materials and notes?
- What will be revised after the first use?

## Chapter Map

You will build the workflow audit in stages:

1. Separate a teaching system from a list of tools.
2. Notice where scattered files and hidden decisions make a lesson fragile.
3. Practice the chapter method: source, prompt, build, verify, teach, archive, improve.
4. Complete a workflow audit for one mini-unit or recurring teaching routine.
5. Check whether another educator could understand the workflow without a live explanation.

## Example Workflow

Here is a finished example of the kind of thinking this chapter is building toward.

| Workflow part | Robotics example |
| --- | --- |
| Source | Course standard on collecting and interpreting sensor data |
| Student task | Teams collect five distance readings and decide whether their robot should stop |
| Evidence | Data table, threshold decision, and reflection sentence |
| Teacher material | One-page directions, sensor setup note, sample data table |
| Verification | Teacher checks sensor wiring, vocabulary, and data-table clarity before class |
| Archive | Final version, sample student response, and revision note are stored together |

## Start the Artifact

Create a first row in your **Teacher workflow audit**. Do not try to solve the whole course. Pick one workflow that is small enough to inspect.

Use these fields:

- Workflow name
- Course or class context
- Source or standard
- Student-facing task
- Evidence students produce
- Files or tools used
- Verification step before teaching
- Revision note after teaching

Example first row:

| Field | Entry |
| --- | --- |
| Workflow name | Sensor-threshold mini-unit setup |
| Course context | Grade 8 robotics |
| Source or standard | Students collect data, identify patterns, and justify a design decision |
| Student-facing task | Measure distance readings and choose a stop threshold |
| Evidence | Team data table and design reflection |
| Files or tools | Slide directions, sensor setup note, response sheet |
| Verification | Test sensor reading, check vocabulary, confirm table has units |
| Revision note | Add a sample row if teams confuse centimeters and inches |

## Quality Check

Before moving on, confirm that your first audit row names an actual teaching workflow. It should not say only "use Google Drive" or "make a lesson plan." It should name what the teacher prepares, what students do, what evidence appears, and how the teacher will know the workflow is ready.

## Reflection

What part of your teaching work currently depends on memory, habit, or a private explanation? Write one sentence naming what would need to become visible for another educator to reuse it.
