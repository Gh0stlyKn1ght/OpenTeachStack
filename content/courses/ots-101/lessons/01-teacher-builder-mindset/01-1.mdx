## Why this matters

When teachers run multiple lessons with no shared system, the first sign appears in chaos: different folder names, different rubrics, different ways to give feedback.

In this lesson, **Why Teachers Need Systems** is your practical answer to that recurring pain.
The goal is not polish. The goal is reliability under pressure.

## Classroom scenario

In a blended class, the same team uses three different check-in methods: Google Form, manual notes, and a different sheet template.
Students submit late work because no one knows which version is authoritative, and the teacher wastes extra prep time reconciling everything.

This is where a teacher system prevents repeated rework: it decides who owns what, where it lives, and how it gets reviewed.

## Core idea

A system is a repeatable way to make teaching choices visible:
- where evidence lives,
- who is allowed to edit it,
- and what must be true before students use it.

In this lesson, the goal is to align **Why Teachers Need Systems** to one audience, one artifact flow, and one revision trigger.

## Weak example

Weak pattern:

> “Every teacher should use their own tools, and students will follow any workflow as long as we explain it well.”

This is true in intent but weak in practice: no one can tell what the official workflow is when something breaks.

## Better example

Better version:

> “By Friday, our team will use one system: one central planning folder, one student handoff folder, and one review folder.  
> Any student submission not using these paths is returned for correction before class review.”

It names the rule and makes it enforceable.

## Teacher action (10-30 minutes)

1. Write a one-paragraph "system promise" for this chapter:
   - What do students see?
   - Who reviews it?
   - What happens if it fails?
2. Define three folders/files in your workflow:
   - student-facing folder
   - teacher-review folder
   - archive folder
3. Add one revision rule that blocks inconsistent locations.
4. Add one accessibility check:
   - headings, controls, and task instructions are clear and screen-reader friendly.
5. Add one safety/privacy check:
   - no full names or personal contact details in shared open links.
6. Set a review trigger in the artifact with day/time/owner before the next lesson launch.

## Quality check

- Can another teacher understand the workflow in 60 seconds and start from it?
- Does the system define the review rule before students touch shared files?
- Is there one explicit failure path (what to do if the workflow is broken)?
- Are accessibility and privacy checks specific, not generic?
- Can this be reviewed by a peer without asking the author for backstory?

## Privacy, safety, and accessibility note

Use these lines in your working artifact draft:

- Accessibility: “All instructions use clear action verbs, and all due-time references use both clock time and total minutes.”
- Safety boundary: “Shared workspaces use initials only until explicit family consent allows full names.”

## Reflection question

What one change can you make today so this artifact becomes reusable by another educator without additional clarification?
What one system rule, once removed, is causing the most rework for you today?
