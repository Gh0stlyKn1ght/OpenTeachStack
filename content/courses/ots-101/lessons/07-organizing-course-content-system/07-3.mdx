> OTS-101 frame: AI supports teacher-approved course content that becomes student-facing only after review.

File names are tiny documentation.

A good name helps future you understand the file before opening it.

A bad name creates archaeology.

## The enemy: mystery final

Most teachers have seen some version of this:

```text
lesson.docx
lesson-final.docx
lesson-final2.docx
lesson-final-real.docx
lesson-final-use-this.docx
lesson-ai-fixed.docx
```

Those names are funny until students get the wrong file.

AI makes this worse because drafts multiply quickly.

## Use names with meaning

A useful file name can include:

- course or unit
- lesson number
- artifact type
- short topic
- status
- date when needed

Example:

```text
ots101-lesson-03-student-directions-source-credibility-approved.md
```

That is longer than `lesson-final.md`, but it carries meaning.

## Keep names boring

Boring names are good.

Use lowercase if your system supports it. Use hyphens. Avoid spaces when files may move between systems.

Prefer:

```text
lesson-04-ai-verification-checklist-approved.md
```

Avoid:

```text
AI Checklist NEW FINAL USE THIS!!!.docx
```

The first name is not glamorous, but it will still make sense next month.

## Date exports and archives

Exports can go stale.

Add a date when the file is a copy for a platform or a snapshot.

```text
canvas-lesson-04-ai-verification-export-2026-06-26.md
```

That tells you when it was made and warns you not to treat it as timeless.

## Do not hide status in memory

If the status only lives in the teacher's head, the system will fail when work gets busy.

Use visible signals:

- folder location
- file name
- frontmatter status
- checklist row
- review note

Redundancy is fine when it prevents confusion.

## Simple naming pattern

Use this pattern if you need one:

```text
[course]-[lesson]-[artifact]-[topic]-[status]-[date-if-needed].md
```

Example:

```text
ots101-lesson-05-student-page-evidence-comparison-approved.md
```

<TakeawayStrip tone="teacher">
  File names should reduce cognitive load, not preserve the drama of how the file was created.
</TakeawayStrip>

## Classroom Scenario

A teacher exports a lesson to an LMS, revises the source file the next day, and then forgets which copy has the current directions.

The problem is not laziness. The file names did not carry enough context.

## Weak Version

Weak file names:

```text
lesson-final.md
lesson-final-new.md
lesson-final-use-this.md
```

These names tell a story of panic, not a story of status.

## Better Version

Better file names:

```text
ots101-lesson-05-student-page-evidence-comparison-approved.md
canvas-ots101-lesson-05-export-2026-06-26.md
ots101-lesson-05-ai-draft-needs-review-2026-06-25.md
```

These names show artifact type, topic, status, and when an export was created.

## Build Step

Rename three sample files on paper before touching your real folders:

```text
Old name:
Better name:
What the new name tells future me:
```

## Quality Check

A name is strong when it tells another teacher what the file is, whether it is current, and whether it is source, draft, export, or archive.

## Safety / Accessibility / Source Note

Do not include student names, accommodations, incidents, or private class details in file names. A file name can leak information even if the file itself is protected.

## Reflection

Which file name in your system would confuse you two months from now?

## Capstone Connection

Add the naming pattern to your course content system map. The capstone packet should be easy to navigate before anyone opens a document.

## When This Helps

Use this workflow when your file names rely on memory: final, new, copy, fixed, revised, or use-this. Those names feel fast today and become expensive later.

## Steps

1. Identify the file's course, lesson, or unit.
2. Name the artifact type.
3. Add a short topic phrase.
4. Add the status: draft, needs-review, approved, exported, or archived.
5. Add a date only for exports, snapshots, or archives.
