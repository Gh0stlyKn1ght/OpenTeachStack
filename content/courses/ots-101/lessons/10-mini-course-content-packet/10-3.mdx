Before calling the packet complete, run one final review.

This review is not about catching every typo.

It is about deciding whether the packet is coherent enough to teach from and honest enough to revise.

<FrameworkBlock title="Final review lenses" eyebrow="Capstone check">
  <ConceptCard title="Alignment">
    Target, lesson, task, rubric, and feedback point to the same learning.
  </ConceptCard>
  <ConceptCard title="Student clarity">
    Students can tell what to do, what to use, what to make, and how to check quality.
  </ConceptCard>
  <ConceptCard title="Evidence">
    Student work will show the intended learning, not just activity completion.
  </ConceptCard>
  <ConceptCard title="Safety and access">
    Privacy, accessibility, source quality, copyright, and tool access have been checked.
  </ConceptCard>
  <ConceptCard title="Source control">
    Source, drafts, exports, and archives are labeled clearly.
  </ConceptCard>
</FrameworkBlock>

## Review alignment

Ask:

- Does the learning target name the skill students practice?
- Does the lesson prepare students for that skill?
- Does the assignment require that skill?
- Does the rubric measure that skill?
- Does the feedback help students improve that skill?

If one piece points away from the target, revise it.

## Review student clarity

Ask:

- Can students find the learning target?
- Are materials named clearly?
- Are directions broken into steps?
- Are examples and non-examples visible?
- Is the submission product clear?
- Is the success check easy to use?

If students would need to guess, revise the student-facing page.

## Review evidence

Ask:

- What will students submit or show?
- Can a teacher inspect the target skill in that evidence?
- Does the rubric describe observable qualities?
- Is feedback connected to revision?

If the evidence is weak, the course packet is not ready.

## Review safety and access

Ask:

- Is private student data removed?
- Are links and files accessible?
- Are sources trustworthy and appropriate?
- Are media and excerpts used responsibly?
- Are headings, lists, and navigation clear?
- Are tool instructions accurate?

If the answer is uncertain, mark it as teacher verification needed.

Use OTS-000 notes here when they help: allowed tools, account limits, source links, platform access, backups, secrets, or student-data boundaries. Those notes support the review, but they do not replace the OTS-101 evidence.

## Review source control

Ask:

- Which version is the source of truth?
- Which pieces are drafts?
- Which pieces are student-ready?
- Which pieces are exports?
- Which old versions are archived?

If the source is unclear, stop and fix the system map.

## Final review note

Use this note:

```text
Final course content review:
- Alignment: ready / revise / not ready
- Student clarity: ready / revise / not ready
- Evidence: ready / revise / not ready
- Safety and access: ready / revise / not ready
- Source control: ready / revise / not ready
- Overall packet status: ready for review / revise before review / do not publish
```

<TakeawayStrip tone="teacher">
  The final review protects the course from looking complete before it is actually teachable.
</TakeawayStrip>

## Next step

If any lens came back revise or not ready, that lens is your next action before the packet moves to assembly. Set a review date so the decision does not quietly expire.

If the missing piece belongs to another pathway course, name it as out of scope instead of stuffing it into OTS-101.
