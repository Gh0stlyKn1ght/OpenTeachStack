# Chapter Plan

## Why This Matters

A Google Workspace course is not about collecting more Drive tricks. It is about making teaching work easier to find, safer to share, and simpler to maintain when the school year gets busy.

A workspace system works when a teacher can answer five questions without searching through memory:

- Which file is current?
- Who can view or edit it?
- Where does student evidence go?
- What gets archived after teaching?
- Who maintains the workflow when something changes?

This chapter gives you the first artifact for OTS-201: a **Workspace workflow audit**. The audit helps you inspect one real teaching workflow before you reorganize folders, build templates, or automate anything.

## What You'll Learn

- Tell the difference between a pile of Google files and a dependable workspace system.
- Choose one teacher workflow worth auditing first.
- Identify the active files, student-facing links, evidence locations, access settings, and maintenance routine for that workflow.
- Find the first weak point that makes the workflow hard for another educator to use.
- Start a workspace audit that later chapters will turn into folders, templates, trackers, and a command center.

## Core Idea

Start with the workflow, not the tool.

For this course, a workflow is a repeatable piece of teaching work: launching a unit, sharing lab directions, collecting exit tickets, returning feedback, preparing substitute materials, or archiving a completed project. Google Workspace can support all of those, but only if each file has a clear job.

You are not trying to clean your entire Drive today. You are choosing one workflow and making its current state visible.

## Classroom Scenario

Maya teaches a middle school robotics unit every spring. Her files live in three places: an old "Robotics 2024" folder, a current "Robotics Unit" folder, and a shared department folder. Students use a directions Doc, a slide deck, and a Form exit ticket. The lesson usually works because Maya remembers which links to use.

That memory is the risk. If a substitute or teammate had to run the lesson, they would not know which file is current, where exit-ticket responses land, or which old slide deck should be ignored.

Maya's first audit does not rebuild the workspace. It names the workflow and traces the files.

## Model Artifact or Example

| Audit field | Model entry |
| --- | --- |
| Workflow name | Launch the robotics sensor mini-unit |
| Teaching purpose | Students need current directions, the launch slides, and the exit-ticket form for the first lab day. |
| Active files | `Robotics Sensor Lab - Student Directions`, `Sensor Launch Slides`, `Sensor Exit Ticket Form` |
| Student-facing links | Directions Doc and Form are shared view/respond only through the classroom post. |
| Evidence location | Form responses collect in `Sensor Exit Ticket Responses - Spring 2026`. |
| Owner | Maya owns the workflow; robotics team can comment on teacher materials. |
| Maintenance trigger | After the first lab day, update the directions Doc and archive last year's slide deck. |
| First risk | Two slide decks have similar names, and the older one is still in the active folder. |

## Common Mistake

Unclear audit:

| Field | Unclear draft |
| --- | --- |
| Workflow name | Robotics files |
| Active files | Drive |
| Maintenance trigger | Clean later |

This fails because another teacher cannot act on it. It does not name the current files, the student-facing links, the evidence location, or the decision that needs attention.

## Verification Checklist

- The workflow is a real repeated teaching task, not a whole course.
- The current student-facing file or link is named.
- Student evidence has a clear destination.
- Access is described without exposing private student information.
- The maintenance trigger says when the workflow gets reviewed.
- Another educator could find the current material without asking you to interpret the folder.
