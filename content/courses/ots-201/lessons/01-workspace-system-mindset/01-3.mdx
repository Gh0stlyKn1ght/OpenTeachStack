# Naming, Ownership, and Maintenance

## Why This Matters

Workspace systems fail quietly when names, owners, and maintenance routines are unclear. A file can be accurate and useful, but still cause trouble if no one can tell whether it is current, who may edit it, or when it should be archived.

Good naming is not cosmetic. It is a classroom operations decision.

A workspace system works when file names, owners, and maintenance routines make the current workflow visible before someone has to ask.

## What You'll Learn

- Write file names that show role, unit, audience, and status.
- Assign an owner for a workflow instead of assuming everyone will maintain it.
- Decide when old files leave the active workspace.
- Add naming and maintenance rules to the workflow audit.

## Core Idea

A file name should answer practical questions:

- What is this?
- Who is it for?
- When does it belong?
- Is it current, draft, template, or archive?

Ownership answers a different question: who is responsible for keeping this workflow usable?

Maintenance answers the final question: when does the workflow get checked, revised, or cleaned?

## Classroom Scenario

A ninth-grade computer science team shares files for a web-design unit. The active folder includes:

- `Web Design Slides`
- `Web Design Slides FINAL`
- `Copy of Web Design Slides`
- `Web Design Student Directions`
- `Directions old`

No one is sure which slide deck is current. One teacher updates `FINAL`, another posts `Copy of Web Design Slides`, and students get different directions.

The team does not need a bigger folder structure yet. First, it needs naming, ownership, and maintenance rules.

## Model Artifact or Example

| Rule type | Model rule |
| --- | --- |
| Naming pattern | `[Course] - [Unit] - [Audience] - [Purpose] - [Status]` |
| Student-facing example | `CS9 - Web Design - Student Directions - Current` |
| Teacher-only example | `CS9 - Web Design - Teacher Notes - Draft` |
| Template example | `CS9 - Project Reflection Form - Template` |
| Archive example | `CS9 - Web Design Slides - Archive 2025` |
| Owner | Unit lead updates current files every Friday during the unit. |
| Maintenance trigger | Archive old versions after the final project reflection is collected. |

## Common Mistake

The common mistake is using emotional status labels: `final`, `real final`, `new`, `copy`, `use this`, or `old maybe`. Those labels make sense for one person in one moment. They do not support a shared workspace.

Another mistake is giving everyone edit access because it feels collaborative. Collaboration still needs ownership. Otherwise, no one knows who approves the current version.

## Verification Checklist

- The naming rule avoids `final`, `copy`, and vague date-only labels.
- Student-facing and teacher-only files are distinguishable.
- The workflow has one named owner or role.
- Maintenance has a trigger, not just a wish to clean later.
- Archive language protects old material from being mistaken for current material.
- Permissions match the ownership rule.

## Reflection / Handoff

Ask a colleague to look at one file name from your workflow. Without extra explanation, can they tell whether it is current and who it is for?

Record the answer in your audit.
