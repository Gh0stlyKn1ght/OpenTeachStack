# Chapter Plan

## Why This Matters

A robotics mini-unit can have excellent ideas and still fail in class if the delivery system depends on memory. One missed slide, one stale link, and every class run becomes a rebuild.

In this chapter, the workspace system is the pair of reusable documents that launch and sustain teaching:

- A **student-facing Doc** with task language, evidence instructions, and support links
- A **slide deck** that controls pacing and transitions during launch, work time, and reflection

When those two artifacts become predictable, inspectable, and versioned, a teammate can run a lesson without relying on the original author's memory.

## Chapter Situation

You will design deliverables for this classroom situation:

Grade 8 robotics mini-unit, two-day block, one student-facing goal:

1. Students measure distance with sensors
2. Students decide one safe-stop threshold
3. Students justify the decision with a short design note

Every student sees the same directions, so the Doc must remove ambiguity. The live deck must keep the room in sync, then hand off cleanly to reflection and evidence capture.

## Chapter Map

In this chapter, you will create the core reusable stack for one deliverable pattern:

1. Separate reusable content from delivery pace
2. Build the student-facing Doc template
3. Build the slide delivery rhythm template
4. Add accessibility and student clarity checks
5. Assemble the final reusable package with revision and ownership rules

## Common Mistake

A common failure is treating a polished deck as “ready.” Readiness is earned only when there is a clear:

- current version ID in the filename or header
- explicit evidence path
- explicit support note for what students do if they are blocked
- documented handoff note for who updates what after teaching
