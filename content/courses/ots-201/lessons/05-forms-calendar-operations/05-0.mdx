# Chapter Plan

## Why This Matters

Forms and Calendar are easy to use badly. A teacher can create a quick Form, collect twenty responses, and still miss the next instructional move because no review time, owner, or follow-up action was planned.

A workspace system works when current materials, student-facing links, evidence locations, permissions, and maintenance routines are clear enough for another educator to inspect.

In this chapter, Forms and Calendar become an operations workflow. A Form collects a signal. A response Sheet stores evidence. A Calendar event protects review time. The teacher action closes the loop.

## What You'll Learn

- Decide when a Form should be used for checks, intake, reflection, or logistics.
- Write Form questions that produce useful teacher action.
- Name the response Sheet and review routine before students submit anything.
- Use Calendar to schedule response review, feedback return, and workspace maintenance.
- Build an operations workflow map that another educator could run during a busy week.

## Core Idea

Plan the follow-through before publishing the Form.

Every useful operations workflow has five parts:

| Part | Question |
| --- | --- |
| Trigger | What starts the workflow? |
| Collection | What Form or event captures the signal? |
| Evidence | Where do responses or status notes land? |
| Review | When and by whom will the evidence be checked? |
| Action | What happens because of the evidence? |

## Classroom Example

Maya teaches a robotics sensor lab on Tuesday. She wants students to submit a quick exit ticket before they pack up. The exit ticket should tell her whether students can connect sensor distance data to a design decision.

Weak version:

- Form title: `Exit Ticket`
- Questions: `What did you learn?` and `Any questions?`
- Response Sheet: automatic default name
- Review plan: check later

That workflow depends on memory.

Stronger version:

| Workflow part | Model decision |
| --- | --- |
| Trigger | Students finish the first sensor test. |
| Collection | `Sensor Lab Exit Ticket - Unit 03` |
| Evidence | `Sensor Lab Exit Ticket Responses - Spring 2026` |
| Review | Tuesday 3:10-3:25 PM Calendar event: Review sensor exit ticket |
| Action | Group students by misconception, revise Wednesday warm-up, flag two lab teams for check-in |
| Maintenance | Archive Form at unit closeout and copy one example response pattern into the unit tracker |

## Model Artifact Start

Start the **Operations workflow map** with one workflow that happens more than once.

| Field | Model entry |
| --- | --- |
| Workflow name | Sensor lab exit ticket |
| Classroom purpose | Check whether students can use sensor data to justify a robot stop threshold. |
| Student-facing tool | Google Form linked from the lab directions Doc and classroom post. |
| Evidence location | Named response Sheet in the Unit 03 teacher folder. |
| Review time | Tuesday after school, 15 minutes. |
| Teacher action | Adjust Wednesday warm-up and identify teams needing support. |
| Owner | Maya reviews responses; robotics teammate can comment on the response Sheet. |
| Closeout rule | Move Form and response Sheet to `Unit 03 Archive` after final reflection. |

## Verification Checklist

- The workflow names a real classroom action, not just a tool.
- The Form has a purpose that leads to a teacher decision.
- The response Sheet has a human-readable name.
- Calendar protects review time.
- The next action is specific enough to do.
- The closeout rule prevents old Forms and response Sheets from staying active forever.
