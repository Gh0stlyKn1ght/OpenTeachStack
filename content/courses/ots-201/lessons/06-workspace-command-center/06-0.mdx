# Chapter Plan

## Why This Matters

By this point in OTS-201, you have inspected workflows, folder rules, templates, trackers, and Forms/Calendar operations. The final challenge is making those pieces usable during a real teaching week.

A workspace system works when a teacher can open one home base and find the current unit, active student links, evidence locations, review queue, maintenance tasks, and revision history without searching through memory.

The **Workspace command center** is that home base. It can be a Google Doc, Sheet, Site, or well-structured folder index. The format matters less than the job it does.

## What You'll Learn

- Decide what belongs inside a command center and what should stay linked.
- Create sections for current unit work, student-facing links, tracker status, operations, maintenance, and revisions.
- Add status labels that tell a teacher what needs action.
- Run a peer review to find missing links or unclear ownership.
- Test the command center against a realistic teaching-week walkthrough.

## Core Idea

The command center is not a decorative dashboard. It is an operating surface.

It should answer five questions quickly:

| Question | Command-center evidence |
| --- | --- |
| What am I teaching now? | Current unit section and active lesson links |
| What do students need? | Student-facing Docs, Slides, Forms, and due dates |
| Where is the evidence? | Tracker, response Sheets, assessment notes, and work folders |
| What needs action? | Review queue, blocked items, and maintenance checklist |
| What changed? | Revision log and archive notes |

## Classroom Scenario

Maya is finishing her robotics workspace. She has a Drive folder map, reusable student directions template, unit tracker, sensor exit-ticket workflow, and weekly maintenance routine. But those pieces still live in separate places.

During a busy week, Maya does not want to remember every link. A teammate should also be able to open one command center and see:

- the current Unit 03 folder,
- the student-facing sensor lab directions,
- the exit-ticket Form,
- the response Sheet that needs review,
- the next Calendar maintenance event,
- the revision note from the last class.

## Model Command-Center Sections

| Section | Purpose |
| --- | --- |
| Current unit | Names the active unit, teaching dates, and current folder. |
| Active student links | Holds only the links students should use now. |
| Evidence and tracker | Links to the unit tracker, response Sheets, and assessment notes. |
| Operations queue | Shows Forms, Calendar events, response review, and follow-up actions. |
| Maintenance checklist | Lists daily, weekly, unit-end, and term-end checks. |
| Revision log | Records what changed, why, and who made the decision. |

## Verification Checklist

- The command center is built around teacher action, not decoration.
- Student-facing links are separated from teacher-only materials.
- Every linked artifact has an owner or review routine.
- Old files and closed Forms have an archive path.
- Status labels make next actions visible.
- A colleague could use the command center without a tour.

## Common Mistake

A common mistake in **Chapter Plan** is treating the visible file or activity as proof that the teaching system is ready. The reviewer still needs to see the classroom purpose, the artifact evidence, the privacy or safety boundary, and the next revision trigger.

For this lesson, do not mark **Workspace command center** ready until another educator can explain what changed, why it matters for **Workspace Command Center**, and what would cause the work to be revised.

## Reflection
