# Peer Review and Revision

## Why This Matters

The person who built the workspace is the worst person to judge whether it is obvious. You already know where everything is, so missing labels and hidden assumptions feel invisible.

A workspace system works when another educator can inspect the command center and find the current path without needing a private explanation.

## Core Idea

Peer review should test findability, access, evidence, action, and maintenance.

Do not ask "Does this look good?" Ask the reviewer to complete small tasks.

## Reviewer Tasks

Give a colleague or future-you these tasks:

| Task | What the reviewer should find |
| --- | --- |
| Start tomorrow's lesson | Current student directions, launch deck, and active Form |
| Check student evidence | Current tracker and response Sheet |
| Handle a confused team | Support note or next action in the operations queue |
| Close the week | Archive rule, closed links, and maintenance checklist |
| Understand the last change | Revision log entry with date, decision, owner, and reason |

## Model Review Notes

Maya's teammate reviews the command center and writes:

| Reviewer finding | Revision |
| --- | --- |
| The exit-ticket Form is linked, but I cannot tell if it is active or closed. | Add status label `Active through Friday`. |
| The response Sheet is easy to find, but I do not know when it gets reviewed. | Add Calendar review event link next to the response Sheet. |
| The revision log says "fixed slides," but not why. | Change entry to `Updated sensor data slide after students confused threshold range`. |

Each revision makes the command center more usable.

## Verification Checklist

- The reviewer performs tasks instead of giving general opinions.
- The review includes student-facing links and teacher-only evidence.
- The review checks whether response data leads to action.
- Revision notes name the actual change.
- At least one unclear command-center section is revised.
- No private student information is exposed during review.

## Common Mistake

A common mistake in **Peer Review and Revision** is treating the visible file or activity as proof that the teaching system is ready. The reviewer still needs to see the classroom purpose, the artifact evidence, the privacy or safety boundary, and the next revision trigger.

For this lesson, do not mark **Workspace command center** ready until another educator can explain what changed, why it matters for **Workspace Command Center**, and what would cause the work to be revised.

## Reflection

What did the reviewer ask you to explain? Convert that explanation into a label, status, link note, or revision-log entry.
