# Permissions and Authorization

## Why This Matters

Most classroom failures in automation come from permissions confusion, not broken business logic. If you cannot explain who can run what, the script is not classroom-safe.

## Core Idea

Define and document permission boundaries before adding live actions:

- which identity runs the script
- which folders/files it may write
- which sheet trigger is allowed
- what is blocked in sandbox

## Permission Guard Example

```javascript
function assertSandboxPermissions() {
  const user = Session.getEffectiveUser().getEmail().toLowerCase();
  if (!user.endsWith("@example.edu")) {
    throw new Error("Use a sanctioned Workspace domain for sandbox testing.");
  }

  const root = DriveApp.getFolderById("FOLDER_ROOT_SANDBOX_ID");
  // Throws a readable permission failure if access is missing.
  root.getName();
}
```

## Safe Runtime Check

```javascript
function runAuthorizationCheck() {
  try {
    assertSandboxPermissions();
    Logger.log(JSON.stringify({ level: "info", status: "authorized", step: "permission-check" }));
    return { level: "info", status: "authorized", step: "permission-check" };
  } catch (error) {
    Logger.log(JSON.stringify({ level: "error", status: "blocked", message: error.message }));
    throw new Error("Aborted: update script owner and sandbox folder access, then rerun.");
  }
}
```

## Setup and Expected Behavior

1. Run `runAuthorizationCheck()`.
2. If authorized, check log includes:
   - `level: "info"`
   - `status: "authorized"`
3. If blocked, fix domain/permissions first and rerun before continuing with `createUnitArtifactsSafe`.

## Common Failure Example

`Exception: Unexpected error while getting the method or property getFolderById on object DriveApp.`

Recovery:

- confirm `FOLDER_ROOT_SANDBOX_ID` is typed correctly
- confirm the script owner can access the folder
- confirm the folder is a sandbox test container

## Permission Record in Artifact

| Failure recovery owner | course automation lead |
