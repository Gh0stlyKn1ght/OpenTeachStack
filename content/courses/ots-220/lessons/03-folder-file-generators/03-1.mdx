# DriveApp Concepts

## Why This Matters

Students benefit when each automation creates files predictably. DriveApp code should show what folder it reads, what folder it writes to, and what it does not change.

## Core Idea

Define drive actions explicitly:

- source: one sandbox root
- write target: one unit folder with fixed children
- forbidden: editing active folders or removing existing content

## Classroom Example

For robotics setup:

- Root sandbox: `FOLDER_ROOT_SANDBOX_ID`
- Target unit folder: `Unit 03 - Sensor Lab - Sample Robotics 8A`
- Child folders: `Evidence`, `Resources`, `Teacher Notes`
- Each child folder created only once per run

## Concrete Code

```javascript
const SANDBOX_ROOT_ID = "FOLDER_ROOT_SANDBOX_ID";

function createUnitFolders(unitName, sectionName) {
  const root = DriveApp.getFolderById(SANDBOX_ROOT_ID);
  const unitFolder = root.createFolder(`${unitName} - ${sectionName}`);

  const evidence = unitFolder.createFolder("Evidence");
  const resources = unitFolder.createFolder("Resources");
  const notes = unitFolder.createFolder("Teacher Notes");

  const output = {
    level: "info",
    unitName,
    sectionName,
    unitFolderId: unitFolder.getId(),
    evidenceFolderId: evidence.getId(),
    resourcesFolderId: resources.getId(),
    notesFolderId: notes.getId(),
    status: "folders-created",
  };

  Logger.log(JSON.stringify(output));
  return output;
}
```

## Expected Success Output

`Logger` should include:
- status `folders-created`
- one unit folder id
- three child folder ids
- no `undefined` field values

## Common Failure

`Exception: Unexpected error while calling getFolderById`

Recovery:

1. verify `FOLDER_ROOT_SANDBOX_ID` is correct
2. verify script account can write the root
3. rerun only after cleanup

## Quality Check

- Could a teammate find and rerun the target root and children from the code alone?
- Did we avoid touching unknown folders?
