# Naming and Idempotency

## Why This Matters

Duplicate folders and mixed naming destroy auditability. A safe generator should skip existing outputs when the exact same run exists, or replace only when explicitly reset.

## Core Idea

Use deterministic naming and an idempotent check before creating.

```javascript
function sanitizeFolderName(value) {
  return value.toString().trim().replace(/[\\/:*?"<>|]/g, "-");
}

function buildUnitFolderName(unitName, sectionName) {
  return `${sanitizeFolderName(unitName)} - ${sanitizeFolderName(sectionName)}`;
}

function folderExists(root, folderName) {
  const found = root.getFoldersByName(folderName);
  return found.hasNext() ? found.next() : null;
}
```

## Example Run

```javascript
function buildOrReuseUnitFolder() {
  const config = getSampleUnitConfig();
  const unitName = buildUnitFolderName(config.unitName, config.sectionName);
  const root = DriveApp.getFolderById(SANDBOX_ROOT_ID);

  const existing = folderExists(root, unitName);
  if (existing) {
    Logger.log(JSON.stringify({ level: "info", status: "reused", folderId: existing.getId() }));
    return existing.getId();
  }

  const created = root.createFolder(unitName);
  Logger.log(JSON.stringify({ level: "info", status: "created", folderId: created.getId(), folderName: unitName }));
  return created.getId();
}
```

## Failure and Recovery

- If duplicate name appears unexpectedly:
  - log and reuse explicit existing folder
  - do not create a second copy in the same run
- If naming has invalid chars:
  - sanitize before `createFolder`
- If root access fails:
  - stop and update boundary notes

## Quality Check

1. Same input twice returns the same folder ID (reused).
2. Naming rules are documented in the lesson artifact.
3. A rollback exists if the wrong folder is created.
