# Script Review Checklist

## Why This Matters

A teacher learning small Apps Script automations safely needs more than a tool list or a checklist. The goal in **Teacher Automation Toolkit** is to make a classroom system that can be used, inspected, revised, and handed to another educator without needing a meeting first.

For **Script Review Checklist**, the practical pressure is simple: automating a confusing process before the trigger, input, output, owner, and failure path are clear. This page turns that pressure into a visible update to **Automation toolkit**.

## Classroom Scenario

Use this situation as the working example: a fake-data automation that creates unit folders from a Sheet row, writes a log entry, and can be deleted without touching real class files. By the end, the work should support this habit: test with fake data, inspect the output, and keep a rollback path.

In this scenario, the learner should be able to point to:

- what the teacher prepares before students see the work,
- what students will read, build, submit, or discuss,
- what evidence shows the work is ready,
- what gets revised after the first use.

## Key Idea

Automation is ready only after the manual routine is clear, fake-data tests pass, logs are readable, and rollback is possible.

Watch for this mistake: Running a script on live class data before proving what it creates, skips, changes, and can undo.

## What You Will Do

Make this page concrete by writing three small decisions:

- Before class, connect script review checklist to one setup decision in the automation toolkit: what the teacher prepares, where it lives, and which source or safety check keeps it honest.
- During class, connect script review checklist to one student action in sandbox folders, fake student rows, copied docs, forms responses, reminders, and logs: what learners read, build, submit, discuss, or revise while working through a fake-data automation that creates unit folders from a sheet row, writes a log entry, and can be deleted without touching real class files.
- After class, review script review checklist through the teacher automation toolkit lens: what evidence proves the move worked, which mistake to avoid (running a script on live class data before proving what it creates, skips, changes, and can undo), and what revision supports test with fake data, inspect the output, and keep a rollback path.

Add this to your artifact:

## What Good Work Looks Like

A strong **script review checklist** entry is concrete enough that another teacher can inspect it. It names the class context, the artifact location, the student product, the source or safety check, and the revision trigger.

For this lesson, a strong entry might say:

> Test log: Fake row Unit 03 / Sample Student A created one folder, one copied Doc, and one log row; no email sent during sandbox test.

That level of detail is enough to review. It is also small enough to revise after one classroom use.

## Quality Check

- The next classroom use or revision is obvious.
