# Automation Toolkit

## Why This Matters

Automation is ready when a teacher can point to the decision, the classroom use, the evidence, and the next review step.

In **Teacher Automation Toolkit**, the work sits inside teacher automation with Google Apps Script and Workspace files. The risk is automating a process before the trigger, input, output, owner, permission boundary, and failure path are clear. This lesson narrows that risk into one visible update to **Automation toolkit**.

## Core Idea

Build the **Automation toolkit** as a usable artifact, not just a note. The finished version shows the decision, the classroom context, the evidence, and the next maintenance step.

The goal is not to make a perfect document in one pass. The goal is to make the teacher decision clear enough that another educator can inspect it, reuse it, or challenge it without needing a private explanation.

## Lesson Focus

Because this is an artifact lesson, the teacher turns automation toolkit into something another educator can use. The artifact needs labels, examples, status, review notes, and a next action instead of a vague promise to improve later.

In 06.3, focus on automation toolkit. Choose one concrete classroom material or teacher decision from this list: a spreadsheet row, folder ID, template Doc, script function, or email draft. Connect it to automation toolkit, then decide whether it needs to be created, revised, tested, credited, restricted, published, or archived.

Connect automation toolkit back to teacher automation toolkit. That connection matters because a polished artifact can still fail if the lesson purpose, boundary, evidence, and maintenance routine are not visible.

## Classroom Example

Use this working situation: a teacher wants a script that creates unit folders, generates lesson Docs from Sheet rows, and sends reminder emails only after the teacher has reviewed the recipients.

For **Automation Toolkit**, choose one concrete classroom material or teacher decision from this list: a spreadsheet row, folder ID, template Doc, script function, or email draft. Keep the example small enough to review during the lesson and focused on automation toolkit.

## Build The Work

Update the **Automation toolkit** with one row or section for **Automation Toolkit**.

| Field | Your note |
| --- | --- |

| Teacher decision | What automation toolkit causes you to keep, revise, test, publish, restrict, or remove |
| Artifact status | What proves automation toolkit is ready or what blocks it |
| Maintenance trigger | When the automation toolkit decision should be checked again |

## Quality Check

Before moving on, test the work against this standard: the script can be run on sample data, stopped safely, explained by another teacher, and repaired when an error appears.

A reviewer should be able to find the automation toolkit decision, inspect the automation toolkit evidence, and understand how it supports teacher automation toolkit.

If the work only names a tool or a file, revise it. The artifact needs to show the teaching purpose, evidence, boundary, and maintenance routine.
