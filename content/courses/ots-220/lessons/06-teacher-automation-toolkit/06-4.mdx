# Checkpoint Review

## Why This Matters

This chapter is a launch-readiness decision point.
Readiness is not about having lots of lines of code; it is about having predictable outcomes and clear recovery.

## Readiness Rubric

- **Ready**
  - Manifest, checklist, and runbook exist and are current.
  - Emergency stop blocks sends immediately.
  - Every reviewer-approved send is logged with reviewer and timestamp.
  - No unplanned send path exists.
- **Revise**
  - Missing approval records or unclear stop mechanism.
  - Functions write sensitive data into logs/notes.
  - Send status transitions are inconsistent.
- **Blocked**
  - Missing consent check.
  - No recovery path after failed sends.

## Required Evidence Pack

Capture these five artifacts before leaving 06:

| Artifact | What to capture |
| --- | --- |
| Manifest | component list + owners + last reviewed date |
| Checklist | function-level status for all script functions |
| Runbook | emergency stop and update instructions |
| Execution sample | at least one logged successful send |
| Failure sample | at least one row marked `error` with reason |

## Acceptance Check

| Check | Expected |
| --- | --- |
| `STOP_AUTOMATION = TRUE` | Running batch throws a stop error |
| One approved row | moves to `sent` |
| One non-approved row | remains non-sent |
| One failed send | status `error` and notes reason |

## Final Reflection

Write one paragraph answering:
> If a parent asks how a reminder email was generated and sent, can you prove it with this toolkit?

If the answer is not verifiable from the evidence pack, mark this chapter as `Revise`.
