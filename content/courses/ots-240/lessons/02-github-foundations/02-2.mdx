# README as Teacher Documentation

## Why This Matters

Open work needs when a teacher can point to the decision, the classroom use, the evidence, and the next review step.

In **GitHub Foundations for Teachers**, the work sits inside open curriculum publishing with GitHub, README files, licenses, attribution, and review routines. The risk is treating free-to-view material as automatically open, reusable, remixable, or safe to publish. This lesson narrows that risk into one visible update to **Repository orientation notes**.

## Core Idea

Make the decision readable to someone else. The artifact explains the purpose, the source of truth, the review routine, and how future revisions will be recorded.

The goal is not to make a perfect document in one pass. The goal is to make the teacher decision clear enough that another educator can inspect it, reuse it, or challenge it without needing a private explanation.

## Lesson Focus

Because this is a focused section, the teacher isolates the readme teacher documentation decision from the rest of the course. The lesson asks what this idea changes, where it appears, and how the teacher will know it worked.

In 02.2, focus on readme teacher documentation. Choose one concrete classroom material or teacher decision from this list: a repository file, README section, issue, pull request, source note, or license entry. Connect it to repository orientation notes, then decide whether it needs to be created, revised, tested, credited, restricted, published, or archived.

Connect readme teacher documentation back to github foundations teachers. That connection matters because a polished artifact can still fail if the lesson purpose, boundary, evidence, and maintenance routine are not visible.

## Classroom Example

Use this working situation: a teacher is preparing a curriculum repository with a README, source notes, a license file, contribution guidance, and a release checklist.

For **README as Teacher Documentation**, choose one concrete classroom material or teacher decision from this list: a repository file, README section, issue, pull request, source note, or license entry. Keep the example small enough to review during the lesson and focused on readme teacher documentation.

Example entry:

| Decision | Classroom version |
| --- | --- |
| Teaching purpose | This work supports readme teacher documentation inside github foundations teachers. |
| Evidence | The teacher can point to source notes, license decisions, README sections, review comments, attribution records, and release status. |
| Boundary | The teacher names what readme teacher documentation may change and what must stay protected or reviewed. |
| Maintenance | The next update is triggered by a date, failed check, student confusion, changed source, or repository orientation notes review. |

## Quality Check

Before moving on, test the work against this standard: a reviewer can see what may be reused, what must be credited, what changed, and how the repository will be maintained.

A reviewer should be able to find the readme teacher documentation decision, inspect the repository orientation notes evidence, and understand how it supports github foundations teachers.

If the work only names a tool or a file, revise it. The artifact needs to show the teaching purpose, evidence, boundary, and maintenance routine.
