# Workshop: Review a Small Change

## Why This Matters

Open work needs when a teacher can point to the decision, the classroom use, the evidence, and the next review step.

In **Contribution and Review Workflows**, the work sits inside open curriculum publishing with GitHub, README files, licenses, attribution, and review routines. The risk is treating free-to-view material as automatically open, reusable, remixable, or safe to publish. This lesson narrows that risk into one visible update to **Review log**.

## Core Idea

Complete the workshop with sample or low-risk material first. Record the setup, result, error or review note, and what changed before using it with a live class.

The goal is not to make a perfect document in one pass. The goal is to make the teacher decision clear enough that another educator can inspect it, reuse it, or challenge it without needing a private explanation.

## Lesson Focus

Because this is a workshop, the teacher works with a safe sample before touching live students, public files, or active course systems. The workshop records setup, result, error, revision, and the decision that follows.

In 04.3, focus on small change. Choose one concrete classroom material or teacher decision from this list: a repository file, README section, issue, pull request, source note, or license entry. Connect it to log, then decide whether it needs to be created, revised, tested, credited, restricted, published, or archived.

Connect small change back to contribution workflows. That connection matters because a polished artifact can still fail if the lesson purpose, boundary, evidence, and maintenance routine are not visible.

## Classroom Example

Use this working situation: a teacher is preparing a curriculum repository with a README, source notes, a license file, contribution guidance, and a release checklist.

For **Workshop: Review a Small Change**, choose one concrete classroom material or teacher decision from this list: a repository file, README section, issue, pull request, source note, or license entry. Keep the example small enough to review during the lesson and focused on small change.

Example entry:

| Decision | Classroom version |
| --- | --- |
| Teaching purpose | This work supports small change inside contribution workflows. |
| Evidence | The teacher can point to source notes, license decisions, README sections, review comments, attribution records, and release status. |
| Boundary | The teacher names what small change may change and what must stay protected or reviewed. |
| Maintenance | The next update is triggered by a date, failed check, student confusion, changed source, or log review. |

## Quality Check

Before moving on, test the work against this standard: a reviewer can see what may be reused, what must be credited, what changed, and how the repository will be maintained.

A reviewer should be able to find the small change decision, inspect the log evidence, and understand how it supports contribution workflows.

If the work only names a tool or a file, revise it. The artifact needs to show the teaching purpose, evidence, boundary, and maintenance routine.
