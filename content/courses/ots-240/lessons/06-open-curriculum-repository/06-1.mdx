# Repository Release Checklist

## Why This Matters

Open work needs when a teacher can point to the decision, the classroom use, the evidence, and the next review step.

In **Open Curriculum Repository**, the work sits inside open curriculum publishing with GitHub, README files, licenses, attribution, and review routines. The risk is treating free-to-view material as automatically open, reusable, remixable, or safe to publish. This lesson narrows that risk into one visible update to **Open curriculum repository**.

## Core Idea

Turn the lesson idea into an inspectable decision. Name the classroom use, the evidence, the review step, and the next revision trigger.

The goal is not to make a perfect document in one pass. The goal is to make the teacher decision clear enough that another educator can inspect it, reuse it, or challenge it without needing a private explanation.

## Lesson Focus

Because this is a focused section, the teacher isolates the repository release checklist decision from the rest of the course. The lesson asks what this idea changes, where it appears, and how the teacher will know it worked.

In 06.1, focus on repository release checklist. Choose one concrete classroom material or teacher decision from this list: a repository file, README section, issue, pull request, source note, or license entry. Connect it to open curriculum repository, then decide whether it needs to be created, revised, tested, credited, restricted, published, or archived.

Connect repository release checklist back to open curriculum repository. That connection matters because a polished artifact can still fail if the lesson purpose, boundary, evidence, and maintenance routine are not visible.

## Classroom Example

Use this working situation: a teacher is preparing a curriculum repository with a README, source notes, a license file, contribution guidance, and a release checklist.

For **Repository Release Checklist**, choose one concrete classroom material or teacher decision from this list: a repository file, README section, issue, pull request, source note, or license entry. Keep the example small enough to review during the lesson and focused on repository release checklist.

| Decision | Classroom version |

## Quality Check

Before moving on, test the work against this standard: a reviewer can see what may be reused, what must be credited, what changed, and how the repository will be maintained.

A reviewer should be able to find the repository release checklist decision, inspect the open curriculum repository evidence, and understand how it supports open curriculum repository.

If the work only names a tool or a file, revise it. The artifact needs to show the teaching purpose, evidence, boundary, and maintenance routine.
