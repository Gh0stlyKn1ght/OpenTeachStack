# Repository Release Checklist

## Why This Matters

Publishing only because files look complete is a common failure mode.
Release criteria must be explicit and auditable.

## Release Gate Framework

Use this table as the minimum checklist before public release:

| Gate | Requirement | Evidence |
| --- | --- | --- |
| Content completeness | all course units and required files exist | file list in final review |
| Licensing | top-level license chosen and referenced | `LICENSE` + README section |
| Attribution | all external source assets credited | `SOURCE_NOTES.md` |
| Privacy | no student-identifying info in public content | repo scan + sample row review |
| Review evidence | at least one teammate review entry | review note or PR comment |
| Maintenance | next review date and owner documented | `maintenance.md` or README section |

## Practical Gate Template

```text
Release Gate: OTS-240 final publish
Owner: ____
Review date: ____
Allowed to publish: yes/no
Blocking issues:
1) ____
2) ____
3) ____
```

## Workshop Exercise

Complete a mock gate with your current repo:

- if any gate is `no`, set `Allowed to publish: no`,
- list at least one owner and date,
- archive unresolved items before moving on.

## Release Decision Language

Use exact wording for consistency:
- **Publish**: all gates pass.
- **Publish internally**: all gates pass except public policy or district-linking items.
- **Revise**: at least one mandatory gate fails.
- **Blocked**: student data risk exists.
