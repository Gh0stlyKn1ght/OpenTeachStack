# Maintenance Plan

## Why This Matters

Open repos decay quickly without defined ownership.
This section locks in who updates and how often.

## Maintenance Rhythm

Use this fixed cycle:
- **Weekly:** review open issues and user questions.
- **Monthly:** run release checklist.
- **Each term:** rotate backup owner and refresh links.

## Maintenance Log Table

| Artifact | Owner | Frequency | Trigger | Last updated | Next action |
| --- | --- | --- | --- | --- | --- |
| README | Primary lead | Weekly | PR merged | 2026-09-20 | verify setup steps |
| LICENSE | Backup owner | Monthly | policy change request | 2026-09-20 | confirm terms |
| Source notes | Primary lead | Monthly | new external asset | 2026-09-20 | add source record |
| Open resource decision log | Team | Weekly | confusion in reuse request | 2026-09-20 | add boundary note |

## Common Maintenance Risks

- old links or dead files,
- missing branch after teammate changes,
- forgotten release date,
- stale attribution after media replacement.

## Runbook Snippet

```text
1. Open `maintenance.md`
2. Check all rows in maintenance table
3. Update `last updated`
4. Record if any gate changed
5. Re-run `Repository Release Checklist` if a gate changed
```

## Rollback Rule

If any maintenance check fails with unclear impact:
- freeze publishing,
- revert last change to staging branch,
- mark issue as `blocked` until resolved.
