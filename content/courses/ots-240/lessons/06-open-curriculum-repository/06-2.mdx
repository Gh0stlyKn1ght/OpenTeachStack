# Maintenance Plan

## Why This Matters

Open work needs when a teacher can point to the decision, the classroom use, the evidence, and the next review step.

In **Open Curriculum Repository**, the work sits inside open curriculum publishing with GitHub, README files, licenses, attribution, and review routines. The risk is treating free-to-view material as automatically open, reusable, remixable, or safe to publish. This lesson narrows that risk into one visible update to **Open curriculum repository**.

## Core Idea

Make the decision readable to someone else. The artifact explains the purpose, the source of truth, the review routine, and how future revisions will be recorded.

The goal is not to make a perfect document in one pass. The goal is to make the teacher decision clear enough that another educator can inspect it, reuse it, or challenge it without needing a private explanation.

## Lesson Focus

Because this is a focused section, the teacher isolates the maintenance decision from the rest of the course. The lesson asks what this idea changes, where it appears, and how the teacher will know it worked.

In 06.2, focus on maintenance. Choose one concrete classroom material or teacher decision from this list: a repository file, README section, issue, pull request, source note, or license entry. Connect it to open curriculum repository, then decide whether it needs to be created, revised, tested, credited, restricted, published, or archived.

Connect maintenance back to open curriculum repository. That connection matters because a polished artifact can still fail if the lesson purpose, boundary, evidence, and maintenance routine are not visible.

## Classroom Example

Use this working situation: a teacher is preparing a curriculum repository with a README, source notes, a license file, contribution guidance, and a release checklist.

For **Maintenance Plan**, choose one concrete classroom material or teacher decision from this list: a repository file, README section, issue, pull request, source note, or license entry. Keep the example small enough to review during the lesson and focused on maintenance.

Example entry:

| Decision | Classroom version |
| --- | --- |
| Teaching purpose | This work supports maintenance inside open curriculum repository. |
| Evidence | The teacher can point to source notes, license decisions, README sections, review comments, attribution records, and release status. |
| Boundary | The teacher names what maintenance may change and what must stay protected or reviewed. |
| Maintenance | The next update is triggered by a date, failed check, student confusion, changed source, or open curriculum repository review. |

## Quality Check

Before moving on, test the work against this standard: a reviewer can see what may be reused, what must be credited, what changed, and how the repository will be maintained.

A reviewer should be able to find the maintenance decision, inspect the open curriculum repository evidence, and understand how it supports open curriculum repository.

If the work only names a tool or a file, revise it. The artifact needs to show the teaching purpose, evidence, boundary, and maintenance routine.
