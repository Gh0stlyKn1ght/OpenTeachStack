# Release Checklist

## Why This Matters

A course site is a maintained teaching surface when a teacher can point to the decision, the classroom use, the evidence, and the next review step.

In **Published Course Hub**, the work sits inside course websites, static hosting, course hub structure, accessibility, privacy, and maintenance. The risk is treating a live website as finished when links, privacy, accessibility, source notes, and update routines are still unclear. This lesson narrows that risk into one visible update to **Published course hub**.

## Core Idea

Turn the lesson idea into an inspectable decision. Name the classroom use, the evidence, the review step, and the next revision trigger.

The goal is not to make a perfect document in one pass. The goal is to make the teacher decision clear enough that another educator can inspect it, reuse it, or challenge it without needing a private explanation.

## Lesson Focus

Because this is a focused section, the teacher isolates the release checklist decision from the rest of the course. The lesson asks what this idea changes, where it appears, and how the teacher will know it worked.

In 06.1, focus on release checklist. Choose one concrete classroom material or teacher decision from this list: a page, link, navigation item, resource entry, deployment target, or maintenance task. Connect it to published course hub, then decide whether it needs to be created, revised, tested, credited, restricted, published, or archived.

Connect release checklist back to published course hub. That connection matters because a polished artifact can still fail if the lesson purpose, boundary, evidence, and maintenance routine are not visible.

## Common Mistake

A common mistake in **Release Checklist** is treating the visible file or activity as proof that the teaching system is ready. The reviewer still needs to see the classroom purpose, the artifact evidence, the privacy or safety boundary, and the next revision trigger.

For this lesson, do not mark **Published course hub** ready until another educator can explain what changed, why it matters for **Published Course Hub**, and what would cause the work to be revised.

## Quality Check

Before moving on, test the work against this standard: students can find the current material, sensitive files are not public, links work, and the teacher knows when to update the site.

A reviewer should be able to find the release checklist decision, inspect the published course hub evidence, and understand how it supports published course hub.

If the work only names a tool or a file, revise it. The artifact needs to show the teaching purpose, evidence, boundary, and maintenance routine.
