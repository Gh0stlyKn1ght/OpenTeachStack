# Context, Constraints, and Acceptance Criteria

## Why This Matters

An agent can draft changes when a teacher can point to the decision, the classroom use, the evidence, and the next review step.

In **Prompting Coding Agents**, the work sits inside safe coding-agent workflows for small teacher tools, prompts, diffs, tests, and review evidence. The risk is letting an agent own the instructional decision, overwrite authored content, or treat a passing build as the only proof. This lesson narrows that risk into one visible update to **Safe agent prompt**.

## Core Idea

Turn the lesson idea into an inspectable decision. Name the classroom use, the evidence, the review step, and the next revision trigger.

The goal is not to make a perfect document in one pass. The goal is to make the teacher decision clear enough that another educator can inspect it, reuse it, or challenge it without needing a private explanation.

## Common Mistake

A common mistake in **Context, Constraints, and Acceptance Criteria** is treating the visible file or activity as proof that the teaching system is ready. The reviewer still needs to see the classroom purpose, the artifact evidence, the privacy or safety boundary, and the next revision trigger.

For this lesson, do not mark **Safe agent prompt** ready until another educator can explain what changed, why it matters for **Prompting Coding Agents**, and what would cause the work to be revised.

## Quality Check

Before moving on, test the work against this standard: the teacher can explain the change, verify the behavior, reject unsafe edits, and preserve the learning goal.

A reviewer should be able to find the context constraints acceptance criteria decision, inspect the safe agent prompt evidence, and understand how it supports prompting coding agents.

If the work only names a tool or a file, revise it. The artifact needs to show the teaching purpose, evidence, boundary, and maintenance routine.
