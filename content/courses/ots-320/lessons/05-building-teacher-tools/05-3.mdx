# Review and Revise the Tool

## Why This Matters

A teacher using coding agents while protecting authored curriculum needs more than a tool list or a checklist. The goal in **Building Teacher Tools** is to make a classroom system that can be used, inspected, revised, and handed to another educator without needing a meeting first.

For **Review and Revise the Tool**, the practical pressure is simple: letting an agent rewrite authored curriculum, accepting a passing build as the only review, or testing with real student data. This page turns that pressure into a visible update to **Small classroom tool**.

## Classroom Scenario

Use this situation as the working example: an agent-assisted printable rubric page where the prompt names allowed files, blocked files, acceptance criteria, tests, and teacher review steps. By the end, the work should support this habit: scope the task, review the diff, test with fake data, and keep teacher judgment in control.

In this scenario, the learner should be able to point to:

- what the teacher prepares before students see the work,
- what students will read, build, submit, or discuss,
- what evidence shows the work is ready,
- what gets revised after the first use.

## Key Idea

An agent can draft changes, but the teacher owns scope, review, testing, acceptance, and rollback.

Watch for this mistake: Letting an agent rewrite curriculum content or accepting code because the build passed once.

## What You Will Do

Make this page concrete by writing three small decisions:

- Before class, connect review and revise the tool to one setup decision in the small classroom tool: what the teacher prepares, where it lives, and which source or safety check keeps it honest.
- During class, connect review and revise the tool to one student action in branches, prompts, diffs, tests, fake data, screenshots, rollback notes, and small classroom tools: what learners read, build, submit, discuss, or revise while working through an agent-assisted printable rubric page where the prompt names allowed files, blocked files, acceptance criteria, tests, and teacher review steps.
- After class, review review and revise the tool through the building teacher tools lens: what evidence proves the move worked, which mistake to avoid (letting an agent rewrite curriculum content or accepting code because the build passed once), and what revision supports scope the task, review the diff, test with fake data, and keep teacher judgment in control.

## What Good Work Looks Like

A strong **review and revise the tool** entry is concrete enough that another teacher can inspect it. It names the class context, the artifact location, the student product, the source or safety check, and the revision trigger.

For this lesson, a strong entry might say:

> Agent entry: Allowed files are rubric page and style module; blocked files are lesson MDX; acceptance requires diff review, route smoke, and fake-data check.

That level of detail is enough to review. It is also small enough to revise after one classroom use.
