# Completed Artifact

## Completed Capstone Proposal Sample

This sample shows what a capstone proposal looks like when it is ready for studio review. It uses fictional classroom details and sample-only evidence.

## Proposal Summary

| Field | Completed example |
| --- | --- |
| Capstone title | Robotics troubleshooting mini-unit system |
| Teaching need | Students need a repeatable way to document two sensor test cycles before changing robot code. |
| Learners served | Grade 8 robotics teams and one rotating student support coach |
| Primary artifact | Mini-unit packet with source notes, student test table, troubleshooting rubric, sample data, and revision log |
| Success signal | 80% of teams complete two documented test cycles within one class period |
| Evidence set | System map row, artifact index row, fake-data test table, peer review notes, release checklist |
| Privacy boundary | Public examples use fictional sample rows; real student logs stay private |
| Release decision | Revise before public release until source date and maintenance trigger are visible |
| Maintenance trigger | Lead teacher reviews after first pilot lesson or by September 20 |

## Ready, Revise, Blocked Rows

| Status | Evidence visible | Reviewer action |
| --- | --- | --- |
| Ready | Student test table, sample completed row, rubric line, and peer review note are linked from the artifact index. | Move to system map review. |
| Revise | Source notes exist, but the current source version is not dated. | Add source date and current-version note before release review. |
| Blocked | Example troubleshooting logs include student names and classroom identifiers. | Rewrite as fictional sample data or keep private. |

## Reviewer Notes

- Ready: the teaching need is specific, measurable, and tied to artifact evidence.
- Revise: source version is not dated yet, so a future reviewer cannot tell which activity is current.
- Blocked: any real student troubleshooting logs must remain private or be rewritten as fictional examples.

Use this as a minimum baseline before final presentation. A stronger artifact also includes a one-slide summary, a maintenance owner, and a short note explaining what changed after peer review.
