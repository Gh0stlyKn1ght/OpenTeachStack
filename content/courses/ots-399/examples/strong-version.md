# Strong Version

## Strong Capstone System

| Artifact | Strong evidence |
| --- | --- |
| Capstone proposal | Names teaching need, audience, success signal, privacy boundary, owner, and review trigger. |
| Artifact index | Every artifact has link or location, status, evidence, and reviewer note. |
| System map | Shows source of truth, public/private boundary, and next maintenance trigger. |
| Technical evidence log | Includes fake-data input, output log, failure path, rollback note, and reviewer decision. |
| Release safety review | Checks student privacy, teacher privacy, licensing, accessibility, and link permissions. |
| Peer review log | Records reviewer question, presenter response, decision, and next action. |
| Maintenance plan | Names owner, date, and condition that starts the next review. |

## Strong Release Pattern

A strong capstone does not say "done." It says one of:

- **Publish:** evidence is complete and safe for the intended audience.
- **Publish internally only:** useful for a team but not safe or polished for public release.
- **Revise before publishing:** one fix is required before release.
- **Block publication:** restricted material, unsafe access, or unverified claims must be removed.
