# Choosing a Real Teaching Need

## Why This Matters

A capstone proves the system when a teacher can point to the decision, the classroom use, the evidence, and the next review step.

A capstone only proves readiness when the teaching need is real, measurable, and reviewable.
In **Capstone Scope**, we choose a need that a colleague can evaluate without your private notes.
Without this, final release artifacts become pretty folders instead of a usable system.

## Core Idea

A strong capstone need has:

- a real classroom pressure,
- one measurable outcome,
- one concrete stakeholder, and
- one safety/privacy implication.

The decision should be evidence-driven, not “interesting idea”-driven.

## Classroom Example

### Sample Real Need: Robotics Troubleshooting Mini-Unit

The team sees repeated student issues in the robotics troubleshooting mini-unit:

- Students submit incomplete evidence for failed sensor tests.
- The source activity links are out of date after two months.
- The release notes are ready, but no one has reviewed safety or privacy checks.

This is a real need if each learner can point to one outcome in the first pilot period.

## Proposal Examples

Use this table as a capstone starting point:

| Area | Ready | Revise | Blocked |
| --- | --- | --- | --- |
| Teaching Need | Students revise one robot diagnostic step using a single shared workflow. | Add one more scenario before release. | No reviewer can find the evidence that the workflow is tested. |
| Evidence Type | Unit map, peer review log, technical evidence log. | Test logs need one student edge-case. | Missing privacy boundary for student examples. |
| Review Owner | Lead teacher + one peer reviewer. | Same owner after first revision. | Requires lead owner with safety reviewer sign-off. |
| Release Decision | Release inside pilot with trigger conditions. | Delay release until test pass log is clean. | Remediation required: source and access controls. |

## Reviewer Coaching Notes

When reviewing a capstone need, do not reward ambition by itself. A smaller proposal with clear evidence is stronger than a bigger proposal with hidden decisions.

Useful reviewer comments name the decision and the evidence:

- "Ready: the proposal names the classroom pressure, test evidence, release boundary, owner, and next review date."
- "Revise: the need is real, but the source evidence needs a current-version note."
- "Blocked: the proposed public artifact still includes identifiable student evidence."

## What to Fill in **Capstone proposal**

- **Problem statement (one sentence):** `When students skip setup checks, troubleshooting lessons lose reliability.`
- **Who it serves:** `Grade 8 robotics group and one rotating student support coach.`
- **Success signal:** `80% of students complete two documented test cycles within one period.`
- **Risk register line:** include one row for privacy/accessibility, one for technical, one for maintenance.
- **Evidence links:** proposal link, system map row, artifact index row, peer review notes link.

## Peer Review and Release Check

Before status becomes ready, the proposal must include:

- who owns the rollout date,
- what is still blocked,
- what changes make it revise-worthy,
- and who confirms evidence before public release.

Use these review labels in the proposal grid:

- **Ready**: test logs and review evidence are present.
- **Revise**: one evidence row missing or one risk row unclear.
- **Blocked**: missing permissions, safety, or source-review notes.
