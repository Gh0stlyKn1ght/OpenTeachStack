# Capstone Proposal

## Why This Matters

A capstone proves the system when a teacher can point to the decision, the classroom use, the evidence, and the next review step.

In **Capstone Scope**, the work sits inside capstone assembly across curriculum, automation, publishing, safety, peer review, and final release. The risk is calling the capstone finished because files exist even though reviewers cannot trace the decisions. This lesson narrows that risk into one visible update to **Capstone proposal**.

## Core Idea

Build the **Capstone proposal** as a usable artifact, not just a note. The finished version shows the decision, the classroom context, the evidence, and the next maintenance step.

The goal is not to make a perfect document in one pass. The goal is to make the teacher decision clear enough that another educator can inspect it, reuse it, or challenge it without needing a private explanation.

## Lesson Focus

Because this is an artifact lesson, the teacher turns capstone proposal into something another educator can use. The artifact needs labels, examples, status, review notes, and a next action instead of a vague promise to improve later.

In 01.3, focus on capstone proposal. Choose one concrete classroom material or teacher decision from this list: a capstone artifact, system link, evidence entry, safety review item, presentation slide, or release checklist row. Connect it to capstone proposal, then decide whether it needs to be created, revised, tested, credited, restricted, published, or archived.

Connect capstone proposal back to capstone scope. That connection matters because a polished artifact can still fail if the lesson purpose, boundary, evidence, and maintenance routine are not visible.

## Classroom Example

Use this working situation: a teacher assembles a published curriculum system with linked artifacts, technical evidence, safety review, presentation notes, peer feedback, and maintenance plans.

For **Capstone Proposal**, choose one concrete classroom material or teacher decision from this list: a capstone artifact, system link, evidence entry, safety review item, presentation slide, or release checklist row. Keep the example small enough to review during the lesson and focused on capstone proposal.

Example entry:

| Decision | Classroom version |
| --- | --- |

| Evidence | The teacher can point to proposal rows, system map links, technical evidence, source and license review, peer review notes, release checklist items, and revision logs. |

## Build The Work

Update the **Capstone proposal** with one row or section for **Capstone Proposal**.

| Field | Your note |
| --- | --- |
| Lesson focus | capstone proposal |
| Classroom context | Where capstone scope appears in your course, unit, tool, site, media packet, safety routine, or capstone |
| Working material | Classroom material or teacher decision: a capstone artifact, system link, evidence entry, safety review item, presentation slide, or release checklist row |
| Teacher decision | What capstone proposal causes you to keep, revise, test, publish, restrict, or remove |
| Artifact status | What proves capstone proposal is ready or what blocks it |
| Maintenance trigger | When the capstone proposal decision should be checked again |

## Completed Sample

Use this sample as the level of detail your proposal should reach before peer review.

| Proposal field | Completed example |
| --- | --- |
| Capstone title | Grade 8 robotics sensor mini-unit system |
| Classroom problem | Students can build the robot, but the teacher cannot yet show which source, setup check, student evidence, and revision note make the mini-unit reusable. |
| Primary artifact | Published mini-unit packet with source notes, student directions, response sheet, sample data, rubric, and revision log. |
| Technical evidence | Fake-data test sheet shows three sample distance readings, one unreliable reading, one threshold decision, and a `Logger.log` record from the folder-copy automation. |
| Safety boundary | No real student names, photos, emails, or classroom screenshots appear in the public packet. Sample rows are fictional and marked sample-only. |
| Release decision | Revise before public release. The artifact index is strong, but the maintenance trigger needs an owner and date. |
| Maintenance trigger | Recheck after the first classroom use or by September 15, whichever comes first. Owner: course teacher. |

## Ready, Revise, Or Blocked

| Status | Reviewer evidence | Next action |
| --- | --- | --- |
| Ready | Proposal names the classroom problem, artifact set, safety boundary, evidence, and maintenance owner. | Move into artifact index review. |
| Revise | Proposal has the artifact list but the evidence is vague or the maintenance trigger has no owner. | Add one evidence row and one dated review trigger. |
| Blocked | Proposal depends on real student work, unlicensed media, private files, or tools the reviewer cannot inspect. | Replace with fictional sample data, remove restricted material, or move the item to a private-only boundary. |

## Reviewer Notes

A peer reviewer should leave notes that point to evidence, not preference. Useful notes sound like:

- "Ready: I can trace the mini-unit purpose, sample evidence, safety boundary, and next review date."
- "Revise: The proposal names the automation, but I need the fake-data test output before I can judge the technical evidence."
- "Blocked: The public packet still links to a folder with identifiable student work."

## Quality Check

Before moving on, test the work against this standard: a reviewer can trace the teaching need, inspect each artifact, see the safety checks, and understand how the system will be maintained.

A reviewer should be able to find the capstone proposal decision, inspect the capstone proposal evidence, and understand how it supports capstone scope.

If the work only names a tool or a file, revise it. The artifact needs to show the teaching purpose, evidence, boundary, and maintenance routine.

## Reflection

What would another teacher need to know before trusting this part of the work? Add that missing context to the artifact before you continue.
