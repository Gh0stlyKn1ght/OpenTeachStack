# Release Safety Review

## Why This Matters

A capstone proves the system when a teacher can point to the decision, the classroom use, the evidence, and the next review step.

In **Publishing and Safety Review**, the work sits inside capstone assembly across curriculum, automation, publishing, safety, peer review, and final release. The risk is calling the capstone finished because files exist even though reviewers cannot trace the decisions. This lesson narrows that risk into one visible update to **Release safety review**.

## Core Idea

Build the **Release safety review** as a usable artifact, not just a note. The finished version shows the decision, the classroom context, the evidence, and the next maintenance step.

The goal is not to make a perfect document in one pass. The goal is to make the teacher decision clear enough that another educator can inspect it, reuse it, or challenge it without needing a private explanation.

## Lesson Focus

Because this is an artifact lesson, the teacher turns release safety into something another educator can use. The artifact needs labels, examples, status, review notes, and a next action instead of a vague promise to improve later.

In 04.3, focus on release safety. Choose one concrete classroom material or teacher decision from this list: a capstone artifact, system link, evidence entry, safety review item, presentation slide, or release checklist row. Connect it to release safety, then decide whether it needs to be created, revised, tested, credited, restricted, published, or archived.

Connect release safety back to publishing safety. That connection matters because a polished artifact can still fail if the lesson purpose, boundary, evidence, and maintenance routine are not visible.

## Completed Sample

| Safety field | Completed example |
| --- | --- |
| Public artifact | Robotics troubleshooting mini-unit packet |
| Release surface | Course site page and downloadable sample packet |
| Student privacy check | Public packet uses fictional team names and sample-only rows. Real student work stays in the private evidence folder. |
| Teacher privacy check | Private planning notes, school-only links, and internal comments are not included in the public packet. |
| Copyright/license check | OER table structure includes attribution and license note; blog tip is excluded from student-facing materials. |
| Accessibility check | Student directions have plain-language steps, table headings, and alternate oral explanation option. |
| Security/access check | Public links open without exposing edit access; private folder links are removed from the public index. |
| Release decision | Revise before public release until the resource license note is complete. |
| Maintenance trigger | Recheck after first classroom use, source change, or any public-link permission change. |

## Release Safety Decision Rows

| Status | Evidence | Reviewer action |
| --- | --- | --- |
| Ready | Public packet has fictional samples, visible source/license notes, accessible directions, and no private links. | Approve for limited public release. |
| Revise | Packet is privacy-safe but missing one license note or accessibility fallback. | Add missing note before public release. |
| Blocked | Public packet links to real student work, private school folders, or unlicensed media. | Remove restricted material and run safety review again. |

## Safety Review Notes

- Do not use screenshots that reveal student names, grades, account emails, browser tabs, or school-only links.
- Do not publish a file simply because the link works; check whether the permission level is appropriate.
- Use fictional samples when the learning value is the structure rather than the real classroom record.

## Quality Check

Before moving on, test the work against this standard: a reviewer can trace the teaching need, inspect each artifact, see the safety checks, and understand how the system will be maintained.

A reviewer should be able to find the release safety decision, inspect the release safety evidence, and understand how it supports publishing safety.

If the work only names a tool or a file, revise it. The artifact needs to show the teaching purpose, evidence, boundary, and maintenance routine.
