# How I Built One-Day Lesson Sites

Sometimes tomorrow's lesson has to exist before tomorrow morning.

That does not mean you need a perfect course platform. It means you need a
clear, source-based page that students can use during class: explanation,
visuals, vocabulary, video, task, exit ticket, and source notes in one place.

This is the One-Day Lesson Site Workflow. It is a teacher survival workflow,
not a replacement for full curriculum design.

## Start with the learning need

Before opening Codex, VS Code, Google AI Studio, ChatGPT, Claude, or any other
assistant, write the real classroom need in plain language:

- What do students need to understand today?
- What standards or skills does this connect to?
- What is the one thing students should leave knowing or being able to do?
- What evidence will show they got it?

If you cannot answer those questions, the page will become decoration instead
of instruction.

## Gather trusted sources

A one-day lesson site should still be source-based. Start with the strongest
references available:

- official documentation
- state standards
- university resources
- reputable articles
- open educational resources
- diagrams and videos that directly clarify the concept
- manufacturer or product documentation when teaching tools, software, or hardware

For software topics, official documentation is the first stop. If you teach
Google AI Studio, start with Google's AI Studio quickstart and Gemini prompt
design docs. If you teach GitHub, start with GitHub Docs. If you teach HTML or
CSS, start with MDN Web Docs.

AI can help summarize and structure sources, but it is not the source.

## Extract the teachable structure

Do not paste sources into a page as a wall of information. Pull out the parts
students actually need:

- key idea
- vocabulary
- examples
- non-examples
- diagram or visual
- short explanation
- student task
- exit ticket

This is where the teacher's judgment matters. The goal is not to include
everything. The goal is to build the smallest clear pathway through the idea.

## Build a simple lesson page

Plain HTML and CSS is enough.

A good one-day lesson site can be one page with clear sections, readable cards,
images, videos, vocabulary, and a task. No heavy framework is required. It
should work on the projector and on student devices. It should be simple enough
that you can edit it during prep without fighting the tool.

Use Codex or VS Code as the build assistant:

- Ask for an HTML/CSS structure.
- Ask for teacher-friendly copy.
- Ask for student-facing directions.
- Ask for an embedded video section.
- Ask for a source notes section.
- Ask for an exit ticket.
- Review every source, claim, image, and standard before teaching.

## Embed media carefully

Use YouTube videos only when they directly support the lesson. Use images only
when they clarify the concept. Add attribution or source notes when needed.
Avoid copyrighted images unless usage is allowed.

Do not use AI-generated visuals as factual diagrams without checking them. A
pretty wrong diagram is still wrong.

## Keep it good enough to teach

This workflow works because it accepts the real constraint: class is tomorrow.

Do not chase perfection. Prioritize clarity, engagement, source quality, and the
student task. After class, save the page into the course archive and improve it:

- note what worked
- note where students got confused
- update examples
- replace weak videos
- add missing vocabulary
- link the page into the larger unit later

## Example: What Is a Robot?

Course: Intro Robotics

Output:

- one-page HTML lesson
- embedded short video placeholder
- image or diagram placeholder
- vocabulary cards
- sense-think-act diagram
- student classification task
- exit ticket

Student-facing benefit: the page gives students a single place to find the
explanation, visuals, video, task, and exit ticket instead of jumping between
scattered documents, slides, and links.

Teacher-facing warning: this workflow is useful for daily lesson creation,
emergency lesson building, enrichment lessons, demos, and fast curriculum
prototyping. It should not replace long-term curriculum planning, standards
verification, assessment design, or accessibility review.

<TeacherNote>
The signature move is not "AI makes a lesson." The move is: teacher identifies
the need, gathers trusted sources, uses AI or a coding assistant to build the
page, verifies the result, teaches it, and improves the page after class.
</TeacherNote>
